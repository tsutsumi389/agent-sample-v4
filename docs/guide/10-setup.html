セットアップと運用

このページでは、汎用AIエージェントを自分のマシンで実際に動かすための手順と、運用時に気をつけたい落とし穴をまとめます。前提ツールの準備から make コマンドによる起動、環境変数の意味、よくあるトラブルの対処までを順に説明します。

このエージェントは、ローカルで動く LLM（大規模言語モデル）である Ollama を頭脳に使い、PostgreSQL に会話履歴や記憶を保存するチャット型アプリです。バックエンド（Python + FastAPI）とフロントエンド（React + Vite）の2つのサーバを動かし、別プロセスの Ollama と PostgreSQL に接続します。まずはこの全体像を頭に入れておくと、以降の手順が理解しやすくなります。

構成の詳しい解説は システムアーキテクチャ を、データの保存方法は データモデルと永続化 を参照してください。

前提ツール

動かす前に、次のツールをインストールしておく必要があります。バージョンには理由があるので、特に Node.js と Ollama は条件を満たすものを用意してください。

ツール必要バージョンなぜ必要か
Docker（compose v2）最新推奨PostgreSQL コンテナを docker compose で起動するため。
uv最新推奨バックエンド（Python 3.12）の依存解決と起動に使う高速パッケージ管理ツール。
Node.js20.19+ または 22.12+フロントエンドのビルドツール Vite 8 が要求するバージョン。
Ollama>= 0.11gpt-oss の streaming（逐次出力）と tools（ツール呼び出し）に対応したバージョンが必要。
PostgreSQL イメージpgvector/pgvector:pg17ベクトル検索拡張（pgvector）が同梱されたイメージ。docker-compose.yml で固定指定されている。
⚠️

プレーンな postgres イメージは使えない

長期記憶のベクトル検索のために CREATE EXTENSION vector を実行します。素の postgres イメージにはこの拡張が入っていないため失敗します。必ず pgvector/pgvector:pg17 を使ってください（docker-compose.yml で指定済みなので、通常はそのままで問題ありません）。

クイックスタート

準備が整ったら、リポジトリのルートで次の3ステップを順に実行するだけで起動できます。コマンドは Makefile に定義されているので、細かいオプションを覚える必要はありません。

  1. cp .env.example .env — 環境変数ファイルを用意します。中身は後述の「環境変数(.env)」で説明します。デフォルトのままローカルで動きます。
  2. make setup — 初回セットアップ。PostgreSQL コンテナの起動、Ollama モデルの pull（ダウンロード）、バックエンド・フロントエンドの依存同期をまとめて行います。モデルのダウンロードがあるため初回は時間がかかります。
  3. make dev — PostgreSQL を起動したうえで、バックエンド（:8000）とフロントエンド（:5173）を並列起動します。

起動できたら、ブラウザで http://localhost:5173 を開いてください。フロントエンドの開発サーバが /api へのリクエストをバックエンド（:8000）へ転送するので、画面からそのままチャットを試せます。

ターミナルを分けて個別に起動したい場合は、make dev の代わりに次を使います。

コマンド役割
make backendターミナル1で FastAPI（:8000）を起動。
make frontendターミナル2で Vite 開発サーバ（:5173）を起動。
ℹ️

個別起動でも PostgreSQL は必要

make backend / make frontend は PostgreSQL を自動起動しません。先に make up で PostgreSQL を立ち上げておくか、一度 make dev を使ってください。

セットアップの流れ

make setup と make dev が内部で何を順に実行するのかを図で示します。Makefile では、ターゲットの前提として up や models が連鎖して呼ばれるようになっています。

make dev は、まず PostgreSQL を起動してから、バックエンドとフロントエンドを 並列（make -j2）で立ち上げます。並列なので両サーバが同時に起動し、ログも同じターミナルに混ざって表示されます。

Makefile ターゲット一覧

用意されているコマンドの一覧です。ターゲットを忘れたら make help で同じ一覧を表示できます。

ターゲット内容実体
make helpターゲット一覧を表示Makefile 内のコメントを整形表示
make upPostgreSQL 起動（healthcheck 通過まで待機）docker compose up -d --wait
make downPostgreSQL 停止docker compose down
make models必要な Ollama モデルを pullollama pull gpt-oss / qwen3 / nomic-embed-text
make setup初回セットアップ（up + models + 依存同期）up → models → uv sync → npm install
make devPostgres 起動後、backend + frontend を並列起動up → make -j2 backend frontend
make backendバックエンドのみ起動（依存同期込み）uv sync && uv run uvicorn app.main:app --reload --port 8000

🛑

make clean はデータを消す

make clean は docker compose down -v を実行し、ボリューム（pgdata）ごと削除します。会話履歴も長期記憶もすべて消えるので、本当にリセットしたいとき以外は使わないでください。停止だけなら make down を使います。

環境変数(.env)

.env はバックエンドの設定値をまとめたファイルです。cp .env.example .env で作ったデフォルト値はローカル開発向けに調整済みなので、基本はそのまま動きます。各値の意味は次のとおりです。設定の読み込み方や使われ方の詳細は システムアーキテクチャ を参照してください。

環境変数デフォルト値意味
APP_DATABASE_URLpostgresql://postgres:postgres@localhost:5432/agentPostgreSQL への接続文字列。docker-compose.yml のユーザー/パスワード/DB 名と一致している。
APP_OLLAMA_BASE_URLhttp://localhost:11434Ollama サーバの URL。
APP_CHAT_MODELgpt-ossチャット応答とツール呼び出しを担うメインモデル。
APP_MEMORY_MODELqwen3長期記憶の抽出（メモリ統合）に使うモデル。構造化出力が安定しているため gpt-oss と分けている。
APP_EMBED_MODELnomic-embed-textテキストをベクトル化する埋め込みモデル（768次元）。
APP_NUM_CTX32768gpt-oss のコンテキスト長（一度に扱えるトークン数）。明示が重要（後述）。
APP_REASONING_EFFORTmediumgpt-oss の推論の深さ。
APP_CORS_ORIGINS["http://localhost:5173"]API へのアクセスを許可するオリジン（フロントエンドの URL）。
APP_MCP_CONFIG_PATHmcp_servers.jsonMCP サーバ定義ファイルのパス。
APP_REFLECTION_DELAY_SECONDS30バックグラウンドのメモリ統合を遅延実行するまでの秒数（debounce）。

記憶の仕組みやメモリ統合の流れは 記憶アーキテクチャ で詳しく解説しています。MCP サーバの追加方法は 拡張ガイド を参照してください。

よくある落とし穴・トラブルシュート

起動でつまずきやすいポイントをまとめます。多くは「モデルの状態」と「設定の明示」に関するものです。

🛑

① gpt-oss は必ず再 pull する

2026年2月以前に pull した gpt-oss モデルには、Optional / Union を引数に持つツールを呼び出すと HTTP 500 になるチャットテンプレートのバグがあります。古いモデルを持っている場合は ollama pull gpt-oss で更新してください。make models にもこの pull が含まれています。

⚠️

② Ollama は 0.11 以上が必要

gpt-oss の streaming（逐次出力）と tools（ツール呼び出し）を使うには Ollama が >= 0.11 である必要があります。古い Ollama だとストリーミングやツールがうまく動きません。ollama --version で確認してください。

⚠️

③ PostgreSQL は pgvector イメージ必須

長期記憶で CREATE EXTENSION vector を使うため、プレーンな postgres イメージでは起動に失敗します。pgvector/pgvector:pg17 を使ってください（docker-compose.yml で指定済み）。

⚠️

④ num_ctx を必ず明示する

Ollama のコンテキスト長（num_ctx）はデフォルトが 2048 と小さく、放置するとエージェントの入力が途中で切り捨てられ、応答が途切れたり沈黙したりします。APP_NUM_CTX=32768 のように明示してください（.env.example で設定済み）。

ℹ️

⑤ モデル未 pull は起動時に即座に失敗する

モデルの初期化時に validate_model_on_init=True が効いており、必要なモデルが pull されていないと起動時点で fail-fast（即エラー終了）します。「起動直後に落ちる」場合は、まず ollama list で gpt-oss / qwen3 / nomic-embed-text が揃っているか確認し、足りなければ make models を実行してください。

既知の制約 / v1 スコープ外

現バージョン（v1）はローカルでの単一ユーザー利用を想定したシンプルな構成です。次の制約を理解しておくと運用時に戸惑いません。

  • バックグラウンドのメモリ統合は best-effort — 記憶の抽出・統合はバックエンドのプロセス内（インプロセス）で動きます。プロセスが落ちると、まだ実行されていない保留中の debounce タスクは失われます。
  • シングルユーザー — user_id は API・設定・名前空間まで一貫して配線されていますが、UI は単一ユーザー（固定値 "default-user"）前提です。
  • 認証なし — ローカル開発用途を想定しており、ログインや権限管理はありません。

以下は v1 では 意図的に対象外としている項目です。

非対象の項目補足
認証・認可ログインやアクセス制御は実装しない。
マルチワーカーでの SSE ブロードキャストRedis pub/sub などによる複数プロセス間のストリーム共有は行わない。
ツールのランタイムホットリロードツールや MCP の追加はバックエンド再起動で反映する。稼働中の動的入れ替えはしない。
本番向けリバースプロキシ調整nginx 等での本番チューニングは対象外。
重厚なテストスイートmake test による基本テストはあるが、網羅的なテストは整備しない。
✅

迷ったらまず動かす

細かい設計は後から追えます。まずは cp .env.example .env → make setup → make dev の3コマンドで起動し、http://localhost:5173 でチャットが返ってくることを確認しましょう。うまくいかないときは、このページの「よくある落とし穴」の①〜⑤を上から順にチェックするのが近道です。
