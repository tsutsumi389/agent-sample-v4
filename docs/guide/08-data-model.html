データモデルと永続化

このエージェントが「何を」「どこに」「どんなキーで」保存しているかを整理します。会話の履歴・長期記憶・スレッドの見出し情報という3つの保存先を、テーブル構造とER図つきで初心者にもわかるように解説します。

前提として、このプロジェクトはローカルLLM（Ollama）を頭脳にするチャット型エージェントです。会話の状態管理には LangGraph（エージェントの処理の流れを管理する仕組み）を使い、データは PostgreSQL（pgvector 拡張つき）に保存します。認証はなく v1 はシングルユーザーで、ユーザーは固定値 "default-user"（コード上は user_id）で表されます。永続化レイヤの実装は backend/app/core/db.py に集約されています。

保存するものの全体像

このエージェントが永続化する情報は、大きく次の3種類です。それぞれ役割が異なり、保存先のテーブルも分かれています。

ここで大事なのは、checkpoints と store のテーブルは自分で SQL を書いて作るものではないという点です。これらは LangGraph が用意した setup() という処理が自動でテーブルを作ります（後述）。一方、スレッドの「タイトル」や「作成日時」といったメタ情報は checkpointer の中には含まれていないため、threads という小さなテーブルを1つだけ自前で追加しています。

ℹ️

ポイント

「checkpointer（チェックポインタ）」とは、会話の途中経過（state）をDBに保存・復元する仕組みです。これにより、サーバーを再起動しても会話の続きから応答できます。「store（ストア）」は会話をまたいで覚えておきたい長期記憶の保管庫です。両者は別物で、保存先のテーブルも別です。

データの関係（ER図）

3つの保存先がどう関係しているかをER図で示します。threads と checkpoints は、データベース上の外部キー制約で結ばれているわけではありません。両者は同じ thread_id という値を共有することで論理的に対応づいています。store（長期記憶）はスレッドとは独立しており、user_id 単位で記憶を束ねます。

threads テーブルの定義は db.py の ensure_threads_table() にあり、thread_id を主キーとする5カラム構成です。checkpoints 側の具体的なカラム構成は LangGraph 側が決めるため、図では「会話の state を持つ」という論理的な役割だけを示しています。

長期記憶の namespace（ユーザーごとの分離）

長期記憶は store に保存されますが、その際の住所にあたるのが namespace（名前空間）です。このプロジェクトでは namespace を ("memories", user_id) というタプル（組）で表し、ユーザーごとに記憶を分離します。記憶1件ごとのIDが key（UUID）で、中身が value です。

このアクセスは store_query.py に実装されています。たとえば記憶の一覧取得は次のように namespace を組み立てて asearch を呼びます。

backend/app/memory/store_query.py 
async def list_memories(
    store: BaseStore, user_id: str, query: str | None, limit: int
) -> list[dict]:
    namespace = ("memories", user_id)
    if query:
        items = await store.asearch(namespace, query=query, limit=limit)
    else:
        # クエリ無しは更新日時の降順を保証する (asearch は順序を保証しない)
        items = await store.asearch(namespace, limit=limit + 50)
        items = sorted(items, key=lambda i: i.updated_at, reverse=True)[:limit]

ポイントは、query（検索語）が与えられたときは asearch が埋め込みベクトルによる意味検索を行い、関連度の高い順に item.score つきで返すことです。一方クエリが無いときは順序が保証されないため、いったん多めに取得してから updated_at の降順に並べ替えています。削除は delete_memory() が同じ namespace と key を指定して store.adelete() を呼びます。

value の形は LangMem（記憶の抽出・管理ライブラリ）が決める {"kind": ..., "content": {...}} 形式です。表示用には _to_text() が content を取り出して人間可読なテキストに整形します。記憶がどう作られるかは 記憶アーキテクチャで詳しく扱います。

✅

なぜ namespace にuser_idを入れるのか

設計4本柱の1つ「user_id の貫通」のためです。namespace の2番目に user_id を必ず入れることで、将来マルチユーザー化したときも、あるユーザーの記憶が別ユーザーに混ざらないことが構造的に保証されます。

会話履歴の復元

会話画面を再び開いたとき、過去のやり取りを表示するには checkpointer から messages を取り出す必要があります。この処理が history.py の get_thread_messages() です。LangGraph のエージェントに対し aget_state(config) を呼んで state のスナップショットを取得し、その中の messages をフロントエンドが扱える API 形式へ変換します。

整形は _to_api_message() がメッセージの型ごとに行います。具体的には次のように対応します。

LangGraph のメッセージ型変換後の role付与される主なフィールド
HumanMessageusercontent, id
AIMessageassistantcontent, id, （ツール呼び出しがあれば）tool_calls
ToolMessagetoolcontent, id, tool_call_id, name
SystemMessage 等（出力しない）None を返して除外

AIメッセージにツール呼び出しが含まれる場合は tool_calls 配列に id / name / args が入り、その実行結果である ToolMessage には対応する tool_call_id が入ります。この tool_call_id によって「どの呼び出しの結果か」が紐づきます。返り値のフィールド仕様は APIリファレンスを参照してください。

ℹ️

config に user_id を渡す理由

aget_state に渡す config には thread_id だけでなく langgraph_user_id も含めています。これは長期記憶の namespace と一貫して user_id をエージェント処理に貫通させるためです。

スレッドのライフサイクル

1つの会話スレッドは「作成 → 会話のたびに更新 → 削除」というライフサイクルをたどります。この管理は threads.py の各関数が担当します。

各段階の挙動は次のとおりです。

  1. 作成: create_thread() が t_ ＋ UUID の thread_id を発行し、タイトル未指定なら初期値 "新しい会話"（DEFAULT_TITLE）で1行 INSERT します。
  2. 更新とタイトル自動生成: 会話が進むたびに touch_and_title() が updated_at を now() に更新します。このとき、タイトルがまだ初期値 "新しい会話" のままなら、最初のユーザーメッセージ由来の候補に差し替えます。すでに確定済みのタイトルは上書きしません。
  3. 削除: delete_thread() が thread_id と user_id の両方が一致する行を threads から DELETE します。削除できた場合は True を返します。

タイトルの自動生成ルールは derive_title() が定義しています。ユーザーメッセージの先頭1行の先頭40字（TITLE_MAX_LEN = 40）を切り出し、空文字なら初期値に戻します。

backend/app/services/threads.py 
def derive_title(message: str) -> str:
    title = message.strip().splitlines()[0] if message.strip() else DEFAULT_TITLE
    return title[:TITLE_MAX_LEN] or DEFAULT_TITLE
⚠️

削除しても会話履歴（checkpoint）は残る

delete_thread() が消すのは threads テーブルのメタ情報（一覧に出すタイトル等）だけです。checkpointer 側の会話履歴はそのまま残置されます。一覧からは消えますが、DB内のチェックポイントは別途クリーンアップしない限り残る点に注意してください。

setup() の役割と pgvector

checkpoints や store のテーブルは、アプリ起動時に一度だけ setup() を呼ぶことで作成・マイグレーションされます。db.py の init_persistence() がこれを担当します。

backend/app/core/db.py 
checkpointer = AsyncPostgresSaver(pool)
store = AsyncPostgresStore(
    pool,
    index={
        "dims": 768,
        "embed": OllamaEmbeddings(
            model=settings.embed_model,
            base_url=settings.ollama_base_url,
            num_ctx=8192,
        ),
        "fields": ["$"],
    },
)
await checkpointer.setup()
await store.setup()

ここで store には index 設定があり、"dims": 768 は埋め込みの次元数（nomic-embed-text の出力）、"embed" は埋め込み生成に使う Ollama モデル、"fields": ["$"] は value 全体を埋め込み対象にすることを意味します。これにより長期記憶が意味検索可能になります。

🛑

setup() は冪等だが「必須」、pgvector イメージも必須

setup() は何度呼んでも安全（冪等）ですが、呼ばないとテーブルは自動では作られません。起動時に必ず一度実行する必要があります。また store は 768次元のベクトル列を使うため、PostgreSQL に vector 拡張が必要です。CREATE EXTENSION vector が通る pgvector/pgvector:pg17 イメージを使わないと setup() が失敗します。標準の postgres イメージでは動きません。

なお接続は build_pool() が作る単一の AsyncConnectionPool（共有コネクションプール）の上に、checkpointer と store の両方が同居します。プールは autocommit=True で、dict_row（行を辞書で受け取る設定）を使っています。threads テーブルの作成は ensure_threads_table() が同じプールを使って CREATE TABLE IF NOT EXISTS で行います。

まとめ

  • 保存先は3つ: checkpoints（会話履歴/キーは thread_id）、store（長期記憶/namespace ("memories", user_id) ＋ UUID）、threads（スレッドのメタ情報）。
  • checkpoints と store のテーブルは setup() が自動生成。threads だけ自前の最小テーブル。
  • 履歴復元は aget_state で state を取り出し、role 付き API メッセージへ整形（APIリファレンス）。
  • 長期記憶は user_id で分離（記憶アーキテクチャ）。データベースの設置や pgvector の用意は セットアップと運用へ。
