記憶アーキテクチャ

このAIエージェントは「いまの会話の流れ」を覚える短期記憶と、「ユーザーの好みや事実」を会話をまたいで覚える長期記憶の2つを使い分けます。このページでは、それぞれが何のためにあり、どう保存・検索されるのかを、実装に沿って丁寧に解説します。

ℹ️

ポイント

このエージェントは Ollama 上のローカルLLMを頭脳にするチャット型エージェントです。v1 はシングルユーザーで、ユーザーを表す user_id は固定値 "default-user" です。ただしコードは user_id を引数として貫通させる設計になっているため、将来の複数ユーザー化にもそのまま備えられます。詳しくはシステムアーキテクチャを参照してください。

2種類の記憶

人間の記憶に「いま話している内容を覚えておく作業記憶」と「昨日聞いた相手の好みを覚えている長期記憶」があるように、このエージェントも記憶を2層に分けています。役割がまったく違うため、保存場所も保存単位も別々です。

短期記憶長期記憶
覚えるものいまの会話のやりとり（発言の履歴）ユーザーの好み・事実・嗜好
保存の単位会話スレッド（thread_id）ごとユーザー（user_id）ごと
会話をまたぐかまたがない（その会話だけ）またぐ（別の会話でも思い出せる）
仕組みcheckpointer = AsyncPostgresSaverStore = AsyncPostgresStore + ベクトル検索
検索方法そのスレッドの履歴をそのまま読み込むキーワード一覧 / 意味の近さ（セマンティック検索）

ここで出てきた用語を補足します。checkpointer（チェックポインター）は、会話の途中経過をデータベースに保存しておく仕組みです。Store（ストア）は、会話とは独立した「メモ帳」のような保存領域で、ここに保存した記憶はベクトル（数値の並び）に変換され、意味の近さで検索できます。短期記憶側の詳細はデータモデルと永続化でも扱います。

長期記憶のハイブリッド方式

長期記憶は、性格の異なる2つの方法を組み合わせて作られています。片方だけでは「会話中にユーザーが明示した好みをすぐ反映する」ことと「会話全体をあとからじっくり振り返って事実を抽出する」ことを両立できないため、両方を使います。これらはどちらも LangMem というライブラリの機能で実現しています。

方式いつ動く誰が動かす使う部品
① hot-path ツール会話の最中（即時）エージェント自身が必要に応じてツールを呼ぶcreate_manage_memory_tool / create_search_memory_tool
② バックグラウンド統合チャットのターン完了後（少し遅れて）システムが自動でスケジュールcreate_memory_store_manager + ReflectionExecutor

① は会話の流れの中でエージェントが「これは覚えておこう」「あれって何だっけ」と判断して実行するため、結果がツールの実行ログやメモリパネルにすぐ現れます（ツールの可視化についてはストリーミング設計を参照）。② はユーザーを待たせずに、会話全体を後から振り返って事実を抽出します。役割が補完的なので、両方を併用します。

会話中の記憶（hot-path）

hot-path（ホットパス、＝ユーザーを待たせる「主経路」）では、エージェントに記憶ツールを持たせます。実装は backend/app/memory/tools.py の langmem_hotpath_tools() で、create_manage_memory_tool（保存・更新・削除用）と create_search_memory_tool（検索用）に加え、後述する build_forget_tool（スコープ付き一括忘却用）の3つを返します。いずれにも同じ namespace と store を渡します。

backend/app/memory/tools.py
MEMORY_NAMESPACE = ("memories", "{langgraph_user_id}")

def langmem_hotpath_tools(store: BaseStore) -> list[BaseTool]:
    return [
        create_manage_memory_tool(namespace=MEMORY_NAMESPACE, store=store),
        create_search_memory_tool(namespace=MEMORY_NAMESPACE, store=store),
        build_forget_tool(store),  # スコープ付き一括忘却 (2段階・確認ゲート付き)
    ]

ここで namespace（ネームスペース＝記憶の置き場所を区切る名札）に "{langgraph_user_id}" というプレースホルダが入っている点が重要です。これは実行時に、その会話のユーザーID（langgraph_user_id）へ自動で置き換わります。つまり同じツール定義でも、保存先は常に「そのユーザー専用の棚」になります。

このように、保存も検索もエージェントの「ツール呼び出し」として行われるため、ユーザーは「いま記憶が保存された／参照された」という動きを画面で確認できます。

バックグラウンド統合の流れ

hot-path だけだと、エージェントがツールを呼び忘れた事実は記憶に残りません。そこで、チャットのターンが終わるたびに、会話全体を裏で振り返って記憶を抽出する仕組みを併用します。これが backend/app/memory/manager.py の役割です。

中心となるのが build_reflection_executor() です。ここでは抽出用のLLMとして qwen3（設定値 settings.memory_model）を num_ctx=8192・temperature=0・reasoning=False で用意し、それを create_memory_store_manager(..., enable_inserts=True, store=store) に渡してマネージャを作ります。enable_inserts=True は「新しい記憶の追加（insert）を許可する」設定です。最後にそのマネージャを ReflectionExecutor（後追いで非同期実行する実行器）でラップして返します。

backend/app/memory/manager.py
extraction_model = ChatOllama(
    model=settings.memory_model,   # qwen3
    base_url=settings.ollama_base_url,
    reasoning=False,
    num_ctx=8192,
    temperature=0,
)
manager = create_memory_store_manager(
    extraction_model,
    namespace=MEMORY_NAMESPACE,
    enable_inserts=True,
    store=store,
)
return ReflectionExecutor(manager, store=store)

実際のスケジュールは schedule_reflection() が行います。ターン完了時に executor.submit({"messages": messages}, ..., thread_id=thread_id, after_seconds=after_seconds) を呼び、「いまから after_seconds 秒後に、このスレッドの会話から記憶を抽出してね」と予約します。

ここで効いているのが debounce（デバウンス＝連続入力をまとめる仕組み）です。submit に同じ thread_id を渡すと、まだ実行されていない前回の予約がキャンセルされ、最後の1回だけが走ります。ユーザーが立て続けに発言しても、抽出処理が何度も無駄に走ることを防げます。

⚠️

渡すのは「ターンの差分」ではなく「スレッド全体」

debounce でキャンセルされたターンの内容が失われないよう、schedule_reflection() のコメントにある通り、呼び出し側はそのターンの差分ではなくスレッドの全会話メッセージを渡します。こうすれば、最後に1回だけ実行されても、会話の全内容から漏れなく抽出できます。

ℹ️

なぜ gpt-oss ではなく qwen3 で抽出するのか

会話の本体（チャットとツール呼び出し）は gpt-oss が担いますが、記憶抽出だけは qwen3 を使います。LangMem の記憶抽出は内部で trustcall による構造化出力（決まった形のJSONを返させる処理）に依存しますが、これが gpt-oss + Ollama の組み合わせでは不安定でした（langchain#33116）。そのため、構造化出力を安定して返せる qwen3 を抽出専用モデルに割り当てています。モデルの役割分担ははじめにも参照してください。

メモリパネル用の取得

画面のメモリパネルに記憶の一覧を表示したり、キーワードで絞り込んだりするための問い合わせは backend/app/memory/store_query.py にまとまっています。中心は list_memories() です。

この関数は namespace = ("memories", user_id) を組み立て、引数 query の有無で動きを変えます。

  • query がある場合: store.asearch(namespace, query=query, limit=limit) を呼び、意味の近い記憶（セマンティック検索の結果）を返します。各記憶には類似度を表す score が付きます。
  • query がない場合: store.asearch(namespace, limit=limit + 50) で多めに取得したあと、コード側で updated_at（更新日時）の降順に並べ替えて先頭 limit 件を返します。asearch は並び順を保証しないため、新しい順を保証するためにこの後処理を入れています。

取得した各記憶は、key / content / namespace / updated_at / score を持つ辞書に整形されます。content は _to_text() ヘルパで人間が読める文字列に変換されます。これは LangMem が記憶を {"kind": ..., "content": {...}} のような入れ子の形で持つことがあるため、content キーを再帰的にたどってテキスト化するためです。なお updated_at は _iso_z() で UTC の ISO 形式（末尾 Z）に揃えられます。

記憶の削除は delete_memory() が担当し、store.adelete(("memories", user_id), key) で指定した key の1件を消します。これらの関数がフロントのAPIから呼ばれ、フロントエンドのメモリパネルに反映されます。1件ずつではなく「仕事の話は全部忘れて」のようにまとめて忘れたいときは、次に説明するスコープ付き一括忘却を使います。

長期記憶を忘れる（スコープ付き一括忘却）

「覚える」だけでなく「忘れる」のも長期記憶の大事な機能です。ユーザーが「私の住所は忘れて」「仕事の話はもうしないで」と言ったとき、関連する記憶を意味検索でまとめて消せる必要があります。ただし削除は不可逆なので、雑に自動削除すると大事な情報まで巻き込んで消してしまいます。そこでこのエージェントは、誤削除を防ぐために2段階の確認ゲートと削除後の検証を挟む設計にしています。

処理は backend/app/memory/forget.py（副作用のない純粋関数）と backend/app/memory/store_query.py（Store への I/O）に分かれています。純粋関数に切り出すことで、削除という危険な処理の判断ロジックを単体テストしやすくしています。

段関数役割
① 候補抽出search_forget_candidates()クエリに意味の近い記憶を検索し、key/content/score 付きの候補にする
② 選別select_candidates()最大件数（既定 25 件 = DEFAULT_MAX_KEYS）でクリップ。暴走的な大量削除を防ぐ主たる安全弁。スコア閾値は任意の補助フィルタで既定は無効
③ 削除batch_delete_memories()承認された key 群を一括で adelete。単一ストアなので孤児は出ない
④ 検証verify_forgotten()削除後に各 key を aget で直接照会し、まだ取得できる（＝消えていない）記憶を leaked_keys として報告する

この4段は2つの入口から使われます。エージェント主導では、会話中にエージェントが forget_memories ツール（build_forget_tool）を呼びます。このツールは confirm=false なら候補一覧を返すだけで1件も削除しません。ユーザーに「これらを削除します。よろしいですか？」と提示して明示的な承認を得てから、同じクエリを confirm=true で呼んで初めて削除します。UI 主導では、メモリパネルの「まとめて忘れる」ボタンが POST /api/memory/forget/preview（候補取得・非破壊）→ チェックで対象選択 → POST /api/memory/forget/confirm（実削除＋検証）の順で同じ4段を辿ります（API / フロントエンド参照）。

ℹ️

なぜ「忘却曲線（減衰）」を採用しないのか

記憶の忘却というと、人間のエビングハウスの忘却曲線のように「時間が経つと自動で薄れて消える（time-based decay）」方式を思い浮かべるかもしれません。設計にあたり最新動向を調査しましたが、2025–2026 時点の本番システムでこの自動減衰を採用する例は限定的でした。理由は、減衰モデルは安全上重要な情報（アレルギー・約束・禁止事項など）まで誤って消してしまうリスクがあり、研究段階の手法が多いためです。本番の主流は、本実装のような明示的な削除＋確認＋検証と、会話後のバックグラウンド統合による上書き（暗黙的な更新）の組み合わせです。そこで本エージェントは、複雑で危険な自動減衰は採らず、「ユーザー/エージェントが意図して忘れる」確実な経路に振り切っています。

⚠️

確認なしの削除は禁止（不可逆）

forget_memories ツールのプロンプトには「承認なしに confirm=true を呼んではならない」と明記しています。UI も必ず確認モーダルを挟みます。削除は復元できないため、確認ゲートは設計上の必須要素です。検証で verified=false（残存の可能性）が返ったときは、フロントが警告を表示します。

namespace 設計

長期記憶のすべての保存・検索・削除は、("memories", user_id) という2要素の namespace を通して行われます。先頭の "memories" は記憶の種類を表す固定の名札で、2つ目の user_id によってユーザーごとに棚が完全に分かれます。これにより、あるユーザーの記憶が別のユーザーから見えることはありません。

⚠️

namespace はバイト単位で一致させる

tools.py のコメントにある通り、MEMORY_NAMESPACE は manager.py / store_query.py とバイト単位で一致させる必要があります（langmem#140）。hot-path とバックグラウンド統合はプレースホルダ "{langgraph_user_id}" 付きの定数 MEMORY_NAMESPACE を共有し、メモリパネル取得側は実際の user_id を埋めた ("memories", user_id) を使います。書き込み先と読み出し先がズレると記憶が見つからなくなるため、ここは慎重に揃えています。

限界（明記）

バックグラウンド統合は便利ですが、無制限に信頼できる仕組みではありません。設計上の限界をはっきり書いておきます。

⚠️

バックグラウンド統合は best-effort（ベストエフォート）

ReflectionExecutor は同じプロセス内のバックグラウンドスレッドで動きます。after_seconds 秒の遅延中（debounce 待ち）にプロセスが落ちると、その保留中タスクは失われます。schedule_reflection() の例外処理もログを残すだけで、再実行は保証しません。つまりバックグラウンド統合は「できる範囲で頑張る（best-effort）」ものであり、確実に記憶される保証はありません。確実に残したい情報は、hot-path の保存ツールで会話中に明示的に保存するのが安全です。

この best-effort な性質は、設計の4本柱のうち「シンプルさ」を優先した結果でもあります。外部のジョブキューを持ち込まず、インプロセスで完結させることで構成を単純に保っています。記憶を確実に残す経路（hot-path）と、ベストエフォートで補強する経路（バックグラウンド）の二段構えにすることで、シンプルさと実用性のバランスを取っています。次はストリーミング設計で、これらのツール実行がどう画面に流れるかを見ていきましょう。
