拡張ガイド（ツール / MCP）

エージェントの「できること」を増やすとき、頭脳の制御コード（graph.py など）には一切手を入れません。このページでは、ツール（エージェントが呼べる道具）を ファイルを1つ置くだけ・MCPサーバを JSONに1行足すだけで追加できる仕組みと、その具体的な手順を初心者向けに解説します。

このページの前提

本プロジェクトは、ローカルで動くLLM（Ollama 上の gpt-oss など）を頭脳にしたチャット型の汎用AIエージェントです。ここでいう「ツール」とは、LLM が自分で判断して呼び出せる小さな関数のこと（例: 電卓・Web検索・現在時刻）。LLM は「いま電卓を使うべきだ」と判断すると、ツールを呼んで結果を受け取り、答えに反映します。

本プロジェクトの4つの設計の柱のうち、最も重要なのが 拡張性 です。能力を増やすたびに中核コードを書き換えていると、変更が事故を生み、テストも壊れます。そこで本設計は「中核は固定したまま、外側を足すだけ」で機能が増える形にしています。全体像はシステムアーキテクチャも参照してください。

2つの拡張ポイント

能力を増やす入口は、はっきり2種類だけです。どちらを使う場合でも、エージェント本体の組み立てコードを編集する必要はありません。

拡張ポイント追加方法向いている用途反映タイミング
ネイティブツールbackend/app/tools/ に Python ファイルを1つ置くPython で完結する処理（計算・整形・自前API呼び出しなど）再起動
MCPサーバbackend/mcp_servers.json に設定を1つ足す外部プロセス・他言語・公開されている既製サーバの利用再起動
✅

どちらを選ぶ？

「自分で短い Python を書けば済む」ならネイティブツール。「すでに世の中にあるサーバ（GitHub操作・時刻・ファイルアクセス等）を借りたい」なら MCP。MCP は Claude Desktop と同じ形式の設定なので、既存の MCP サーバをそのまま流用できます。

ツール自動登録の仕組み

ネイティブツールが「ファイルを置くだけ」で有効になるのは、backend/app/tools/registry.py が app/tools/ ディレクトリを自動でスキャンしているからです。スキャンは pkgutil.iter_modules（パッケージ内のモジュール一覧を取得する標準機能）で行われ、各モジュールの中に置かれた module-level の BaseTool インスタンス（@tool で作られたツールオブジェクト）を見つけて登録します。

下の図は discover_package() の探索の流れです。仕組み自体には何も改造を入れず、ただファイルを増やすだけで点線部分（あなたの新しいツール）が自動で取り込まれます。

ポイントを設計事実として整理します。

  • 除外モジュール: registry / base / __init__ の3つはスキャン対象から外れます（_EXCLUDED_MODULES）。レジストリ自身や基底定義をツールと誤認しないためです。
  • 登録対象: モジュールの中で isinstance(obj, BaseTool) が真になるオブジェクトだけを登録します。@tool を付けた関数はこの BaseTool インスタンスになります。
  • 名前の重複は即エラー: register() は同じ tool.name が既にあると ValueError("duplicate tool name: ...") を投げます。気づかないうちに同名ツールが2つできる事故を、起動時に必ず検出します。
  • サードパーティ製も探索: discover_entry_points() が importlib.metadata.entry_points(group="agent.tools") を読み、pip でインストールされた別パッケージのツールも取り込みます（こちらは source="plugin"）。プラグインのロードに失敗しても例外をログに出して読み飛ばし、全体は止めません。
ℹ️

source の意味

登録時に各ツールには出所ラベル（native / plugin）が付きます。MCP 由来は mcp、記憶ツールは langmem として後段で合流します。この出所は GET /api/tools の一覧で確認できます（後述）。

同梱されているサンプルツール

実装の手本として、最初から3つのネイティブツールが app/tools/ に置かれています。いずれも @tool 付き関数で、docstring（関数の説明文）がそのまま「このツールが何をするか」の説明として LLM に渡ります。

ツール名ファイル引数戻り値（要約）
calculatetools/calculator.pyexpression: str数式を安全に評価した結果の文字列（四則演算・べき乗・剰余）
current_datetimetools/datetime_tool.pyなし現在の日時（JST）を「年月日 時刻 + 曜日」で返す
web_searchtools/web_search.pyquery: strDuckDuckGo で検索した上位最大5件のタイトルと概要（APIキー不要）

たとえば電卓ツールは、危険な式の実行を避けるため eval() を使わず、ast（Python の構文解析）で式を読み取り、許可した演算子だけを使って計算します。これは「外部から渡された文字列を安全に評価する」良い例です。

ネイティブツールを足す手順

新しい道具を増やす一番手軽な方法です。やることは「ファイルを作る」「関数に @tool を付ける」「再起動する」の3つだけ。レジストリ側のコードは触りません。

  1. backend/app/tools/ の中に新しいファイル（例: weather.py）を作ります。ファイル名は registry / base / __init__ 以外なら自由です。
  2. その中で @tool を付けた関数を module level に 定義します。引数には型ヒントを付け、docstring に「何をするツールか」を日本語で簡潔に書きます。この docstring が LLM への説明になるため、丁寧に書くほど呼び出しの精度が上がります。
  3. バックエンドを再起動します。起動時に discover_package() がファイルを見つけ、関数を自動登録します。
  4. GET /api/tools で一覧に出ているか、あるいはチャットで実際に呼ばれるかを確認します。

最小の実装例です（既存サンプルと同じ書き方）。@tool を付けた時点で関数は BaseTool インスタンスになり、自動登録の対象になります。

backend/app/tools/weather.py 
"""サンプル: 自作ツールの最小例。"""

from langchain_core.tools import tool


@tool
def get_weather(city: str) -> str:
    """指定した都市の天気を返す。city には都市名を渡す。"""
    # 実際にはここで外部APIなどを呼ぶ
    return f"{city} は晴れ、気温 22℃ です。"
⚠️

名前の衝突に注意

ツール名（@tool 付き関数名、または明示指定した名前）が既存ツールと重複すると、起動時に ValueError: duplicate tool name が出てエージェントが立ち上がりません。これは「壊れた状態で気づかず動き続ける」ことを防ぐ、わざとの安全装置です。名前は他とかぶらない具体的なものにしましょう。

MCPローダーの仕組み

MCP（Model Context Protocol、外部のツール群を標準化された方法で呼び出すための取り決め）を使うと、自分で Python を書かなくても、世の中に公開されている既製サーバの機能をツールとして取り込めます。設定の入口は backend/mcp_servers.json ただ1つで、これは Claude Desktop と同じ {"mcpServers": {...}} 形式です。

読み込みは backend/app/mcp/loader.py が担当します。次の図がその処理の流れです。

設計事実として押さえるべき挙動は次のとおりです。

  • ファイルが無ければ何もしない: mcp_servers.json が存在しなければ空の設定を返し、MCP無しでそのまま起動します。現在の同梱ファイルは {"mcpServers": {}} で、デフォルトでは MCP サーバは登録されていません。
  • シークレットの環境変数置換: 設定値の中の ${ENV_VAR} という記法は、起動時に同名の環境変数の値へ置き換わります（_substitute_env）。トークンなどの秘密情報を JSON に直書きせず済みます。なお該当する環境変数が無い場合は空文字に置換されます。
  • transport の自動補完: 設定に transport が無くても、command があれば stdio（子プロセスとして起動）、url があれば streamable_http（HTTP接続）を自動で補います。どちらも無ければ ValueError で設定ミスを知らせます。
  • 名前衝突の回避: クライアントは MultiServerMCPClient(config, tool_name_prefix=True) で作られます。tool_name_prefix=True により、各ツール名にサーバ名のプレフィックスが付くため、別々のMCPサーバが同名ツールを持っていてもぶつかりません。
  • ステートレスな取得: ツール一覧は get_tools() を呼んで取得します。状態を持たないため、必要なときに集めるだけのシンプルな構造です。

MCPサーバを足す手順

MCP サーバの追加は、backend/mcp_servers.json の mcpServers にエントリを1つ書き加えるだけです。コア側の変更はありません。

  1. backend/mcp_servers.json を開き、mcpServers オブジェクトの中に新しいキー（サーバ名）を追加します。サーバ名は分かりやすい短い名前にします（ツール名のプレフィックスになります）。
  2. 子プロセス型のサーバなら command と args を、HTTP型のサーバなら url を指定します（transport は省略可。前述のとおり自動補完されます）。
  3. トークンなどの秘密情報は値に直書きせず、"${ENV_VAR}" の形で書き、実際の値は環境変数で渡します。
  4. バックエンドを再起動すると、ローダーが新しいサーバのツールを取り込みます。

下は記述例です（time は子プロセス型 stdio、github は環境変数でトークンを注入する例）。

backend/mcp_servers.json 
{
  "mcpServers": {
    "time": {
      "command": "uvx",
      "args": ["mcp-server-time"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}

HTTP 型のサーバを使う場合は、command の代わりに url を書きます。transport は streamable_http として自動補完されます。

{
  "mcpServers": {
    "remote-tools": {
      "url": "https://example.com/mcp"
    }
  }
}
ℹ️

${GITHUB_TOKEN} はどこで設定する？

JSON に書いた ${GITHUB_TOKEN} は、バックエンドのプロセスに環境変数 GITHUB_TOKEN として渡しておくと、起動時にその値へ置き換わります。設定方法はセットアップと運用を参照してください。

起動時のツール集約

ここまでの2つの拡張ポイントは、エージェント起動時に1か所へ合流します。ネイティブツール、サードパーティプラグイン、長期記憶用のLangMemツール、そして MCP ツールを集めて、最終的に create_agent（LangGraph のエージェント生成）へまとめて渡します。中核コードは「集めたツールを受け取る」だけで、何が増えたかを知る必要はありません。

集約された結果は API から確認できます。GET /api/tools を呼ぶと、各ツールに source（native / plugin / langmem / mcp）が付いた一覧が返ります。出所ラベルは describe() が {"name", "description", "source"} の形で組み立てています。どのツールが何由来で有効になっているかを一目で確認できるので、追加後の動作チェックに使えます。詳細はAPIリファレンスを参照してください。

記憶（LangMem）まわりの詳しい仕組みは記憶アーキテクチャ、ツール呼び出しを含む応答がどう画面へ流れるかはストリーミング設計で扱います。

設計上の注意

最後に、拡張するときに知っておくべき制約をまとめます。

  • ホットリロードは無し: ツールや MCP の追加・変更は 再起動で反映されます。実行中のプロセスに動的に差し込む仕組みはありません。落ち着いて「ファイルを直す → 再起動 → /api/tools で確認」の順で進めましょう。
  • 名前は必ずユニークに: ネイティブツールはツール名が重複すると起動時に ValueError で止まります。MCP はサーバ名プレフィックスで衝突を避けますが、サーバ名どうしは分かりやすく区別できる名前にしてください。
  • docstring が説明文: ネイティブツールの docstring はそのまま LLM への説明になります。曖昧だと LLM が呼び出しを誤るため、用途と引数の意味を簡潔に書きましょう。
  • コアは無改修が原則: graph.py など中核を編集せずに能力を増やせるのが本設計の狙いです。拡張のたびに中核を触りたくなったら、それは拡張ポイントの使い方を見直すサインです。
🛑

秘密情報を JSON に直書きしない

API トークンやパスワードを mcp_servers.json に生の文字列で書くと、リポジトリに残って漏えいの原因になります。必ず "${ENV_VAR}" 記法と環境変数を組み合わせてください。
