ストリーミング設計

チャットの応答を「全部できあがってから一気に」ではなく、生成しながら少しずつ画面に届ける仕組み（SSE）を解説します。LangGraph のエージェントが流す途中経過を、どうやって SSE のイベントに変換しているかを、実装に沿って読み解きます。

このページの前提です。本プロジェクトは Ollama（ローカルで動く LLM 実行基盤）を頭脳にしたチャット型エージェントで、応答は POST /api/chat/stream という1本の API から流れてきます。サーバ側は FastAPI（Python の Web フレームワーク）、エージェント基盤は LangGraph（複数の処理ノードをグラフとしてつなぐライブラリ）です。応答を生成しているのは複数のエージェント（オーケストレーター・プランナー・エグゼキューター・レスポンダーなど）で、その全体像はマルチエージェント設計を参照してください。このページでは「その途中経過をどう画面に届けるか」だけに集中します。

SSEとは

SSE は Server-Sent Events（サーバ送信イベント）の略です。ふつうの HTTP 通信は「リクエストを1回送ると、レスポンスが1回返ってきて終わり」ですが、SSE では 1つの HTTP レスポンスをサーバ側が開いたまま、イベントを少しずつ何度も送り続けます。クライアントはそれを受け取るたびに画面を更新できるので、LLM が単語（トークン）を生成するそばから、リアルタイムに文字が増えていくチャット体験になります。

ブラウザには SSE 専用の EventSource という標準 API がありますが、本プロジェクトでは使っていません。理由はシンプルで、EventSource は GET しかできず、リクエストボディを送れないからです。チャットでは「どのメッセージを」「どのスレッドで」送るかを POST のボディ（JSON）で渡す必要があります。そこでフロントエンドは fetch + ReadableStream（受信データを少しずつ読むための仕組み）で POST しつつ、返ってくる SSE の本文を自前で1行ずつ解釈します。フロント側の受信処理はフロントエンド設計で詳しく扱います。

ℹ️

ポイント

サーバ側は FastAPI の EventSourceResponse（SSE を返す専用レスポンス）を使います。15秒ごとの keep-alive コメント（接続が切れていないか確認するための空メッセージ）は FastAPI のルーティング層が自動で挿入してくれるので、アプリ側のコードでは書きません。

チャット1回の全体シーケンス

ユーザーが1回メッセージを送ってから応答が流れ終わるまでの全体像です。POST /api/chat/stream のエンドポイントは backend/app/routers/chat.py にあり、本体のストリーミング変換は backend/app/services/streaming.py の stream_agent() が担います。

重要なのは、SSE の本文を1バイトも送る前にスレッドの存在チェックを済ませる点です。スレッドが見つからなければ 404 を返しますが、これは FastAPI の依存（Depends）として SSE 開始前に走るため、ストリームが始まったあとに途中でエラーになるのではなく、最初のレスポンスとして素直に 404 を返せます。スレッドは事前に POST /api/threads で作っておく必要があります（APIリファレンス参照）。

タイトルの確定タイミングにも工夫があります。stream_agent() はストリームを始める前に touch_and_title() でスレッドのタイトルを決めます。タイトルはユーザーのメッセージだけから決まるので、途中で生成を止められても（stop しても）タイトルだけはきちんと残るようにしているのです。

ストリームの3モードとイベント対応

本プロジェクトの心臓部は、LangGraph のエージェントを次のように呼ぶ箇所です。stream_mode に3つのモードを同時指定し、subgraphs=True でサブグラフ（responder や executor といった内部のミニグラフ）内のイベントまで受信します。

backend/app/services/streaming.py 
async for part in agent.astream(
    {"messages": [{"role": "user", "content": message}]},
    config={...},
    context=AgentContext(user_id=user_id),
    stream_mode=["messages", "updates", "custom"],
    subgraphs=True,
):

subgraphs=True を付けると、astream が返す各要素は (ns, mode, chunk) の3つ組（タプル）になります。ns は「どのサブグラフ由来か」を示す名前空間、mode は上の3モードのどれか、chunk がデータ本体です。コードでは念のため、3要素でないとき（subgraphs=False 相当）にも備えて ns = () として扱う防御的な分岐を入れています。

この3つの mode が、それぞれ別の SSE イベントに変換されます。

stream_modechunk の中身変換後の SSE イベント意味
messages(AIMessageChunk, metadata)tokenLLM が生成したテキストの断片（ユーザー向けノード由来のみ）
updatesノード名→更新内容の dicttool_call / tool_resultツール呼び出しの開始と、その結果
custom任意の dict（status を含む）progress計画・ステップ進捗などの自由な進捗通知

SSE として実際に送られるイベント名と data（JSON）の形は次のとおりです。data の中身は streaming.py の ServerSentEvent(...) 呼び出しがそのまま反映されます。

eventdata の形送出条件
token{"content": "文字断片", "node": "responder"}ユーザー向けノードの AIMessageChunk に空でないテキストがあるとき
tool_call{"id": ..., "name": ..., "args": {...}}AIMessage.tool_calls を id で重複排除して初出のとき
tool_result{"id": ..., "name": ..., "content": "結果(最大2000文字)"}ToolMessage を id で重複排除して初出のとき
progress{"status": "...", ...}custom イベントを受信したとき（必ず status を含む）
done{"thread_id": ..., "title": ...}ストリームが正常に最後まで流れたとき（最後に1回）
error{"message": "エラー文言"}途中で例外が発生したとき
ℹ️

ポイント

tool_result の content は TOOL_RESULT_MAX_CHARS = 2000 で先頭2000文字に切り詰められます。検索結果のような長大なツール出力で SSE が膨れ上がるのを防ぐためです。

マルチエージェント対応の要点

このプロジェクトでは複数のエージェントが内部で会話しながら答えを作ります。そのまま全部を token として流すと、ユーザーには見せたくない「内部の思考（オーケストレーターの判断やプランナーの計画立案）」まで画面に漏れてしまいます。streaming.py は、これを防ぎつつ重複も除くために、いくつかの仕掛けを持っています。

1. トークンの二重防御フィルタ

ユーザーに見せてよいトークンは、最終的な回答を書くノードからのものだけです。コードでは USER_FACING_NODES = {"responder", "synthesizer"} と定義し、これ以外のノード（orchestrator / planner / evaluator / executor などの内部思考）からのトークンは token イベントにしません。

ただしフィルタは1段だけではありません。一次防御はモデル側にあります。内部思考ノードの LLM 呼び出しには tags=["nostream"] が付いていて、そもそもトークンが発生源で抑制されます。streaming.py 側の USER_FACING_NODES 判定は、それをすり抜けたものを止める二次防御です。2段構えにすることで、内部のつぶやきがユーザーに漏れる事故を確実に防ぎます。

2. 重複排除（dedupe）

updates モードでは、同じ情報が2回届くことがあります。responder サブグラフのメッセージは、サブグラフ内の updates と、親グラフレベルの updates（全履歴の再掲）の両方で届くためです。そこで親レベルの responder 更新はスキップし（ns == () and node_name == "responder" の条件）、実体はサブグラフ内から拾います。

さらに tool_call は tool_calls の id、tool_result は tool_call_id をキーに、seen_tool_calls / seen_tool_results という集合で初出かどうかを判定し、2回目以降は送りません。これでツール関連イベントが二重に画面表示される事故を防ぎます。

3. executor 内部の扱い

エグゼキューター（実際にツールを動かす実行担当）の中で起きる tool_call / tool_result は、ユーザーへの進捗としては SSE に流します。一方で、後段の記憶アーキテクチャで使うリフレクション（会話を振り返って長期記憶を抽出する処理）の対象には含めません。コードでは accumulate = ns == () or top == "responder" という条件で、リフレクション用の turn_messages に積むのは親レベルと responder 由来のメッセージだけに限定しています。executor の中間メッセージはステップごとの「下書き（スクラッチパッド）」なので、記憶に残すべき確定情報ではないからです。

4. progress は必ず status を持つ

custom モードの dict は progress イベントとして透過しますが、規約として必ず status（文字列）を含むようにしています。chunk に status が文字列で入っていればそのまま、無ければ {"status": str(...)} の形に整えてから送ります。こうしておくと、古いフロントエンドでも data.status だけ読めば従来どおり進捗を表示できます。

SSEワイヤ形式の例

実際にネットワークを流れるバイト列の例です。SSE では event: 行でイベント名、data: 行で本文（このプロジェクトでは1行の JSON）を送り、空行で1イベントの区切りとします。: で始まる行はコメント（keep-alive）なので、受信側は無視します。

event: progress
data: {"status": "計画を立てています"}

event: tool_call
data: {"id": "call_abc123", "name": "web_search", "args": {"query": "東京 天気"}}

event: tool_result
data: {"id": "call_abc123", "name": "web_search", "content": "東京は本日晴れ ..."}

event: token
data: {"content": "東京", "node": "responder"}

event: token
data: {"content": "の天気は", "node": "responder"}

: keep-alive

event: token
data: {"content": "晴れです。", "node": "responder"}

event: done
data: {"thread_id": "th_001", "title": "東京の天気"}
✅

読み方のコツ

1イベント＝「event: 行 ＋ data: 行 ＋ 空行」の3点セットです。フロント側は受信バッファを空行で分割し、各ブロックを event 名で振り分けて処理します。: で始まる行が来ても何もしないでよいだけなので、接続維持の仕組みは受信ロジックを複雑にしません。

エラー時の挙動

stream_agent() 全体は try / except / finally で囲まれています。ストリーム中に何らかの例外が発生すると、except 節がログを記録したうえで event: error を1回送り、そのままストリームを閉じます。つまり、エラーは黙って切れるのではなく、必ず error イベントとしてクライアントに伝わってから終了します。フロントはこれを受けて「エラーが起きました」と表示できます。

⚠️

done と error は排他

正常に最後まで流れたときだけ done を送ります（completed = True のあと）。途中で例外になった場合は done ではなく error が送られます。クライアントは done か error のどちらかが来たらストリーム終了、と判断できます。

さらに finally 節では、正常完了したターンに限って（completed が真で、かつメッセージが2件以上あるとき）記憶のリフレクションをスケジュールします。リフレクションには今ターンの差分だけでなく、エージェントの最終状態（スレッドの全会話メッセージ）を agent.aget_state() で取り直して渡します。これは thread_id 単位の debounce で保留中タスクがキャンセルされても内容が失われないようにするためです。この記憶の更新フローは記憶アーキテクチャで詳しく解説します。

まとめ

このページでは、LangGraph の astream が返す (ns, mode, chunk) の3タプルを、token / tool_call / tool_result / progress / done / error という SSE イベントに変換する流れを見ました。設計の柱は、ユーザー向けノードだけをトークン化する二重防御フィルタ、id ベースの重複排除、必ず status を持つ進捗イベント、そして エラーも必ずイベントで伝えることです。全体構成はシステムアーキテクチャ、内部のエージェント分担はマルチエージェント設計、API の正確な仕様はAPIリファレンスを参照してください。
