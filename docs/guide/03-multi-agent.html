マルチエージェント設計

このプロジェクトの核心です。ローカルLLM（手元のPCで動かす言語モデル）の性能に頼りきらず、長く複雑なタスクでも最後まで完走させるために、6つの役割に分けたエージェントと「コードが決める」制御の仕組みを解説します。

ℹ️

ポイント

このページだけで読めるよう前提を補います。本エージェントは Ollama（ローカルでLLMを動かす実行環境）上の gpt-oss（会話・ツール実行担当）と qwen3（判断・分類担当）を使い分けます。全体は LangGraph（処理を「ノード」と「エッジ」のグラフで記述するフレームワーク）で組まれています。アーキテクチャ全体像はシステムアーキテクチャを参照してください。

なぜマルチエージェントなのか

1つのLLMに「最初から最後まで全部やって」と丸投げすると、短い質問なら問題なく答えられますが、複数の手順を要する長いタスクでは途中で目的を見失ったり、同じツール呼び出しを延々と繰り返して終わらなくなったりします。ローカルで動かせるサイズのモデルは、特にこの「迷子」が起きやすいという弱点があります。

そこでこのプロジェクトでは、仕事を 役割ごとの小さなエージェント（ノード）に分割 し、さらに 「次に何をするか」「もう終わってよいか」という制御の判断をLLMではなくコード側に持たせる という設計を採っています。LLMには「考える」ことだけを任せ、「進める・止める・戻る」の交通整理は決定論的なPythonの純関数（同じ入力なら必ず同じ結果を返す関数）が担当します。これにより、LLMの出力がどれだけ崩れても、有限のステップで必ず回答にたどり着く（停止性が保証される）構造になっています。

✅

設計の合言葉

「賢さはLLMに、確実さはコードに。」ルーティング・ループ・終了判定は routing.py の純関数が決め、各ノードは失敗しても必ず正常な値を返す（例外を漏らさない）よう作られています。

グラフの全体像

処理の入口は START で、そこから必ず orchestrator（オーケストレーター） に入ります。orchestrator はユーザーの要求を見て、2つの道のどちらに進むかを振り分けます。短い質問なら 高速パス（responder が即答）へ、複数手順が必要そうなら 計画パス（planner → executor → evaluator のループ）へ進みます。

図のうち、矢印の分岐（条件付きエッジ）はすべて routing.py の関数が決めます。graph.py では add_conditional_edges で「orchestrator の後は route_after_orchestrator」「evaluator の後は route_after_evaluation」と紐づけており、ここにLLMは一切登場しません。

6つのノードの役割

各ノードは graph.py の build_agent() 内で生成され、それぞれ得意なモデルを割り当てられています。gpt-oss は会話とツール実行に強いモデル、qwen3 は短い判断（分類・計画・評価）を任せるモデルです。

ノードモデル役割
orchestratorqwen3（control）ターン開始時にスクラッチ（後述）をリセットし、要求を DIRECT（高速パス）か PLAN（計画パス）かに分類する。短い入力はLLMを呼ばず即 direct。
respondergpt-oss（chat）高速パス担当。全ツール＋LangMem（長期記憶）＋全会話履歴を使う ReAct（考える→ツールを使う→また考える、を繰り返す方式）エージェント。単一エージェント時代と同じ即答役。
plannerqwen3（control）ゴールを実行可能なステップの列に分解し、JSON で返す。全履歴は渡さず、ゴールとツールカタログだけを見る。
executorgpt-oss（chat）依存（depends_on）が解決済みのステップ群を max_parallel_executors 件まで asyncio.gather で並列実行する。各ステップは新しいスクラッチパッドで動き、会話スレッドには永続化しない（checkpointer=False）。
evaluatorqwen3（control）そのラウンドで実行されたステップの結果群を並列に判定し（各ステップ pass / retry / replan）、リトライ・再計画の予算管理（上限超過時の格下げ）も行う。
synthesizergpt-oss（chat）全ステップの結果を統合して最終回答（AIMessage）を作る。失敗時は結果を機械的に連結して必ず何か返す。
ℹ️

「responder」がいるのに6つ？

ブリーフでは orchestrator / planner / executor / evaluator / synthesizer / responder の6ノード構成と説明しています。responder は高速パス専用の即答エージェントで、計画パスの executor とはツール・履歴の扱いが異なります（responder は全履歴あり、executor は履歴なし）。

1ターンの流れ（計画パス）

複数手順が必要な要求の場合、planner が計画（各ステップに depends_on 付き）を立て、executor と evaluator がラウンド単位で回ります。1ラウンドでは、依存が解決済みのステップ群を executor が並列実行し、evaluator がその結果群を並列判定します。合格したステップは done になり、その完了で依存が解けた次のステップ群が次ラウンドで走ります。互いに独立なステップは同じラウンドで同時に実行され、依存（DAG）のあるステップだけ順序が守られます。実行可能なステップがなくなったら synthesizer が結果をまとめます。

重要なのは、executor の各ステップは 「まっさらなスクラッチパッド」で動く 点です。graph.py で checkpointer=False として作られているため、他のステップのツール往復のログを引きずらず、与えられるのは「最終目標」「依存ステップ（depends_on）の結果」「今回のタスク」だけ。並列実行では「直前まで」という逐次の前提が成り立たないため、履歴は依存関係で明示された先行ステップの結果に限定します。これによりコンテキストが膨らみすぎず、各ステップに集中できます（詳しくは後述の状態管理を参照）。

高速パス（DIRECT）

「こんにちは」「ありがとう」のような短い挨拶や、ひとことで答えられる質問にまで計画を立てるのは無駄です。orchestrator はそうした入力を direct と分類し、responder に即答させます。さらに router_skip_under_chars 文字未満の極端に短い入力は、LLMでの分類すらスキップして即 direct とし、レイテンシ（応答までの待ち時間）をゼロに近づけます。

responder は単一エージェント時代と同じ ReAct エージェントそのもので、親グラフと messages チャネル（会話履歴の入れ物）を共有し、checkpointer も親から継承します。つまり高速パスは「マルチエージェント以前の、シンプルな1エージェント挙動」へ安全に縮退した状態とも言えます。

評価ループと終了保証

計画パスのループが暴走しないよう、evaluator の判定（verdict）と routing.py の純関数が二人三脚で「必ず止まる」ことを保証します。verdict には4種類あります。

verdict意味次の行き先
passステップ成功。status を done にする（その完了で依存が解けたステップが次ラウンドの対象に）次ラウンド（executor）／実行可能ステップなしなら synthesizer
retry結果が不十分。status を pending に戻し、ステップ専用の feedback を添えて再実行次ラウンド（executor）で再実行
replan計画自体が筋悪。立て直す（needs_replan を立てる）planner（再計画）
fail予算超過などで諦め、status を failed にして前進次ラウンド／synthesizer

停止性（いつか必ず終わること）は、次の多層の安全装置で保証されています。

  • 大域上限が最優先: executor は1ラウンドで実行したステップ数ぶん executor_runs を増やします（並列でも累計でカウント）。route_after_evaluation は他のどの判定よりも先にこれを見て、max_executor_runs に達したら無条件に synthesizer へ抜けます。executor は実行可能なステップがある限り必ず1件以上実行するため、ループは必ず前進します。
  • 実行可能ステップの枯渇: 依存が解決済みの pending ステップが1つも無くなれば synthesizer へ抜けます。循環依存（planner で sanitize_dependencies が壊す）や依存先が failed で詰んだステップは ready にならないため、ここで自然に停止します。
  • retry の上限: 同じステップの試行回数 attempts が max_step_retries を超えたら、evaluator が verdict を fail に格下げします。
  • replan の上限: replan_count が max_replans 以上なら、同じく fail に格下げします。
  • 同時実行数の上限: 1ラウンドで同時に走る executor は max_parallel_executors 件までに絞られ、ローカルLLM（Ollama）への同時リクエスト集中を防ぎます。
  • 親グラフの recursion_limit: さらに保険として、LangGraph の親グラフ全体に graph_recursion_limit が設定されています（streaming.py で渡され、config.py が予算から下限を自動補正）。並列化でラウンド数はステップ実行回数以下に収まるため、逐次時代の導出式がそのまま上界として安全です。executor 内の ReAct にも executor_recursion_limit という別の上限があります。
⚠️

予算管理はノード、ルーティングは純関数

「retry/replan を fail に格下げする」判断は evaluator ノード側で行い、routing.py の純関数には書きません。これにより routing.py は「verdict を見て行き先を返すだけ」の単純な純関数を保てます。役割分担を崩さないことが、デバッグしやすさと停止性の両立につながっています。

モデル非依存の堅牢化（フォールバック）

ローカルLLMは構造化出力（決まった形のJSONを必ず返す機能）が不安定です。そこでこのプロジェクトでは with_structured_output のような仕組みを使わず、LLMにはプレーンテキストで答えさせ、そこから parsing.py がJSONを力技で抽出します。さらに各ノードは「パースに失敗しても必ず妥当な値を返す」フォールバックを持ちます。

ノード失敗時のフォールバック
orchestrator分類に失敗・例外なら direct（高速パス＝既存の単一エージェント挙動へ縮退）
planner計画のパース全滅なら単一ステップ計画 [goal] に縮退（executor 1回の ReAct 実行になる）
executor例外・反復上限なら打ち切りマーカー付きの結果文字列を返して正常リターン（例外を漏らさない）
evaluator空結果・打ち切りはLLMを呼ばず即 retry。パース全滅・例外なら pass（前進フォールバック）
synthesizerLLM失敗・空出力ならステップ結果を機械的に連結して回答（回答ゼロで終わらない）

parsing.py のJSON抽出 extract_json() は、3段構えで頑強です。

  1. <think> の除去: qwen3 などが出す <think>...</think> という思考ブロックを取り除く（strip_think）。
  2. コードフェンス優先: ```json ... ``` のようなコードブロックの中身を最優先で候補にする。
  3. raw_decode 走査: それでも見つからなければ、テキスト中のすべての { の位置から json.JSONDecoder().raw_decode を試し、最初に成功した辞書を返す。全滅なら None。

抽出後は pydantic（データ検証ライブラリ）の PlanSchema / VerdictSchema で型検証します。PlanSchema は steps が文字列のリスト（旧形式）でも、description/step/task/content キーを持つ辞書のリストでも拾えるよう柔軟に作られ、依存も depends_on/deps/after/requires など複数のキー名と数字文字列から正規化します。さらに planner 側の sanitize_dependencies() が自己参照・範囲外・重複・循環依存（Kahn 法で検出）を除去するため、LLM が雑な依存を返しても安全です。さらに parse_with_retry() が「不正なJSONでした、形式だけ出してください」と指示を追記して最大 max_retries 回まで再試行し、それでもダメなら必ず fallback を返します。パースは「絶対に例外を投げない」契約 です。

🛑

構造化出力を使わない理由

trustcall や with_structured_output はローカルLLMでは不安定で、失敗すると例外が外まで漏れてエージェント全体が止まりかねません。「壊れにくいプレーンテキスト出力＋堅牢な抽出＋リトライ＋フォールバック」の方が、モデルの素性に依存せず確実に完走できます。

状態（state）とコンテキスト管理

グラフ全体で持ち回るデータは state.py の AgentGraphState に定義されています。ここで一番大事な原則は、意味的に永続するのは messages（会話履歴）だけ という点です。計画や進捗（plan / executor_runs / needs_replan など）は「ターン内スクラッチ」、つまりその1ターン限りの作業メモで、orchestrator が毎ターン fresh_scratch() で必ず初期化します。これにより、前ターンのチェックポイント（DBに保存された途中状態）に残った古い値が悪さをしないよう、決定論的に上書きされます。

制御フィールドはすべて NotRequired（あってもなくてもよい）で宣言されています。これは、単一エージェント時代の {"messages": [...]} だけの古いチェックポイントもそのまま読めるようにするための後方互換の工夫です。

各ノードが「どこまでの情報を見るか」も注意深く絞られています。これがコンテキスト管理の肝です。

  • responder のみ全履歴を見る: 高速パスの即答役だけが、過去の会話すべてを参照します。
  • planner / evaluator はゴールと結果要約のみ: 全履歴は渡さず、ゴールやステップ結果の要約だけを見るので、判断がぶれにくく軽量です。
  • executor は各ステップ約8K文字以内に切り詰め: _scoped_prompt() が「最終目標」「依存ステップ（depends_on）の結果（executor_history_max_chars の予算内）」「今回のタスク」だけを組み立てます。並列実行では逐次の「直前まで」が成り立たないため、履歴は依存関係で明示された先行ステップの結果に限定します。retry 時にはそのステップ専用の feedback も添えられます。

最後に user_id の貫通 について。ユーザー識別子（v1 では固定値 "default-user"）は2つの経路で同時に渡されます。streaming.py でグラフを呼ぶ際、context=AgentContext(user_id=user_id) として @dataclass のコンテキストを渡すと同時に、configurable にも langgraph_user_id を入れます。

ℹ️

なぜ user_id を「二重に」渡すのか

LangMem の記憶を保存する名前空間（namespace）が ("memories", "{langgraph_user_id}") と定義されており、この {langgraph_user_id} プレースホルダは config.configurable から解決されます。一方、ツールやプロンプトから参照する user_id は AgentContext 経由です。両方をそろえて渡さないと記憶のユーザー分離が崩れるため、context と configurable の双方にプラミングしています。記憶の仕組みは記憶アーキテクチャ、データの永続化はデータモデルと永続化を参照してください。
