チャット内GUI（生成的UI）

本ページは、チャット応答の中にツールが「対話的なUI（テーブル・チャート・フォーム・カード等）」を埋め込む機能、いわゆる生成的UI（Generative UI）の確定仕様です。Claude Artifacts / MCP Apps / OpenAI Apps SDK / Vercel AI SDK の先行事例を調査し、本プロジェクトの「コア（graph.py）無改修でツールを置くだけ」という拡張哲学と、既存のSSEストリーミング契約（未知イベント無視・id dedupe・done/error排他）を一切壊さずに実現する方式へ落とし込みました。

結論を先に述べます。UIリソースは LangChain 標準の ToolMessage.artifact を唯一の封筒（エンベロープ）とし、UIツールは @tool(response_format="content_and_artifact") で書くだけでよい。これにより checkpoint への永続化も自動で乗ります。MVPは「宣言的セーフコンポーネント方式（React側ホワイトリストで描画）」を採用し、任意HTML/JSを動かす「サンドボックスiframe方式」は封筒に mode フィールドを予約するだけで実装は将来に回します。初見の読者でも追えるよう、契約・シーケンス図・コード例・セキュリティ・既存設計書との整合を順に示します。

✅

実装ステータス: MVP（宣言的方式）は実装済み

本仕様の MVP（A 宣言的セーフコンポーネント方式）は実装済みです。バックエンドは backend/app/agent/ui.py（封筒 is_ui_artifact/coerce_ui/ui()）、backend/app/tools/ui_widgets.py（render_table/render_bar_chart/render_card/render_form）、streaming.py（ui_resource emit）、history.py（再水和）、schemas/chat.py（MessageOut.ui）。フロントは frontend/src/components/UIResource.tsx（レジストリ + Table/Chart/Card/Form の各 View、Zod 検証）と、useChatStream.ts/ChatView.tsx/MessageBubble.tsx の配線。B（サンドボックスiframe方式）は封筒の mode 予約のみで未実装です。

ℹ️

前提

本ページはストリーミング設計の SSE 契約と、拡張ガイドの「ツールを置くだけ」哲学を土台にしています。先にその2ページを読むと、ここで「なぜ既存契約を壊さずに済むのか」がより明確になります。

何を解決するのか（背景と原型）

現状のチャットはテキスト（Markdown）とツール呼び出しチップ（ToolCallChip の details 表示）しか描けません。表形式のデータ・グラフ・入力フォームのような「構造を持った情報」も、すべて文字列に潰してバブル内に流すしかありませんでした。生成的UIは、ツールが返す構造化データを、フロントで本物のReactコンポーネントとして描くことでこれを解決します。

先行4事例から抽出した原型は次のとおりです。それぞれ「UIをどう届け」「ホストとUIがどう往復し」「本プロジェクトに何を示唆するか」を整理しました。

事例UIの届け方ホスト⇔UIの往復本プロジェクトへの示唆
Claude ArtifactsLLMが生成したコード塊を別ペインのライブプレビューで描画双方向は薄い（基本表示専用、CSPで網羅遮断）信頼できない生成コードはプロセス分離＋厳格CSPで隔離する
MCP Apps (ext-apps 2026-01-26)ui-scheme のMCPリソース(HTMLテンプレ)を事前宣言、ツールが resourceUri で指す二重iframe + postMessage越し JSON-RPC 2.0content(モデル文脈用テキスト)と structuredContent(UI用データ)の分離思想。未知ホストでもテキストで前方互換
OpenAI Apps SDKmeta openai-outputTemplate がHTML(JSバンドル内包)を指すwindow.openai グローバル注入 + postMessageデータ取得ツールと描画ツールの分離。setWidgetState によるUIローカル状態保持
Vercel AI SDKiframe無し同一オリジン。ツール名→Reactコンポーネント直マップブリッジはストリーム自体（新ツール呼出/メッセージ送信で往復）自前ホストでコンポーネントが信頼できる場合の最有力モデル。input/output 状態遷移が既存 tool_call/tool_result と同型

本プロジェクトには外部ホスト（Claude.ai 等）が存在せず、自前のReact+FastAPIアプリそのものがホストです。したがって、UIを描くコンポーネントは自前資産であり信頼境界の内側にあります。これは Vercel 流の「ツール由来データをクライアントの既定コンポーネントが描く」方式が最も軽く安全に成立することを意味します。RSC版 streamUI が開発停止し「サーバがUIコードを流すのではなくデータを流す」方式が現行推奨になった事実も、この方向性を裏付けます。

✅

重要な割り切り

MVPでは「コンポーネントは自前＝信頼できる」「propsデータだけがツール/LLM由来＝検証対象」という信頼境界を引きます。これにより iframe隔離なしで安全に描画でき、実装が最小になります。

二層アーキテクチャと、MVPの推奨

信頼境界に応じて描画方式を2層に分けます。

（A）宣言的セーフコンポーネント方式 — MVP採用

フロントがホワイトリストレジストリ（table / chart / form / card）を持ち、封筒の component 名で解決して対応Reactコンポーネントを描きます。

  • props は型検証（Zod）を通してから渡す。任意HTML/JSは一切実行しない。XSS面はゼロ。
  • 既存の tool_call（実行中＝input-available相当）→ tool_result（完了＝output-available相当）の2段状態遷移をそのまま流用できる。
  • 未知の component 名は従来の ToolCallChip にフォールバック（前方互換）。

（B）サンドボックスiframe方式 — 封筒に mode を予約、実装は将来

  • mode:"iframe" のとき、任意HTML/JSを sandbox 属性 + CSP + オリジン分離で隔離描画し、postMessage越し JSON-RPC でUI⇔ホストを往復させる（MCP Apps 仕様 2026-01-26 準拠）。
  • 外部MCPサーバが ui-scheme リソースを宣言する将来や、ホワイトリストで賄えない任意UIに備える。

MVPに（A）を強く推奨する理由は4つです。

  1. 信頼境界の一致: 描画コンポーネントは自前資産で信頼できる。隔離が不要なので最も軽い。
  2. 既存契約の再利用: tool_call/tool_result の2段遷移、id dedupe、未知イベント無視がそのまま効く。改修は分岐追加のみ。
  3. ローカル完結・無認証v1との整合: 任意HTML/JS実行はLLM/MCP由来の悪性スクリプトという攻撃面を持ち込むが、無認証ローカルv1では割に合わない。
  4. 段階的拡張: 封筒に mode フィールドだけ予約しておけば、（B）は後から非破壊で追加できる。

UIリソースの契約（封筒の型）

UIリソースは LangChain 既存機構 ToolMessage.artifact を唯一の封筒とします。新しいテーブルも別保存経路も作りません。封筒の型 UIEnvelope は backend/app/agent/ui.py に新設します。

封筒（artifact に格納される dict）の構造は次のとおりです。

キー型必須意味
vint必須スキーマ版（=1）。将来の破壊的変更に備える
kindstr必須固定値 "ui"。これがUI封筒の判定キー
componentstr必須描画コンポーネント名（table/chart/card/form 等）
propsdict必須コンポーネントへ渡すデータ。UI描画用の完全データ
mode"declarative"|"iframe"任意既定 declarative。iframe は将来用に予約
ui_idstr任意指定なければ stream/history 側で tool_call_id を流用

なぜ uri(ui://...) や mimeType を持たないか: MCP Apps は外部サーバのUIテンプレを事前宣言・プリフェッチするため ui-scheme のURIとMIMEが要ります。本プロジェクトは自前ホストでコンポーネントが手元にあるため、URIで「どこから取るか」を指す必要がなく、component 名で「どれで描くか」を指せば十分です。将来（B）で外部MCPの ui-scheme を取り込む際は、その封筒を本封筒へ正規化する変換層（coerce_ui）でURI→component解決を吸収します。

backend/app/agent/ui.py に置く純関数です。streaming.py と history.py の両方から呼び、判定ロジックを一元化（＝DRY）します。

backend/app/agent/ui.py 
UI_SCHEMA_VERSION = 1

def is_ui_artifact(artifact: object) -> bool:
    return (
        isinstance(artifact, dict)
        and artifact.get("kind") == "ui"
        and isinstance(artifact.get("component"), str)
        and isinstance(artifact.get("props"), dict)
    )

def coerce_ui(artifact: object, tool_call_id: str | None) -> dict | None:
    """artifact をUI封筒へ正規化。UI でなければ None。"""
    if not is_ui_artifact(artifact):
        return None
    a = artifact  # type: ignore[assignment]
    return {
        "id": a.get("ui_id") or tool_call_id,
        "component": a["component"],
        "mode": a.get("mode", "declarative"),
        "props": a["props"],
        "v": a.get("v", UI_SCHEMA_VERSION),
    }

判定が artifact ベースであるため、ネイティブツールでもMCPツールでも、artifact が kind=="ui" の dict でありさえすれば同一経路でUI化できます。

ℹ️

ポイント

content(要約テキスト)と props(UIデータ)の分離は MCP Apps の structuredContent/content 分離と同思想です。LLMの後続文脈には content の要約だけが渡り、巨大な props 辞書はコンテキストを汚しません。本プロジェクトの tool_result 2000字制限（ストリーミング設計参照）と完全に同じ「安全とコストの境界」です。

UIツール作成API（コア無改修哲学の維持）

設計原則は「新しいデコレータも基底クラスも発明しない」。LangChain 標準の @tool(response_format="content_and_artifact") をそのまま使わせます。registry.py の自動探索は isinstance(obj, BaseTool) で拾うため、app/tools/ にファイルを置くだけで自動登録され、graph.py もregistryも無改修で動きます。

環境確認済み: backend/.venv は langchain_core 1.4.6 / langchain 1.3.8。response_format="content_and_artifact" は戻り値 (text, artifact) の artifact を ToolMessage.artifact へ格納し、model_dump() でシリアライズ可能（＝checkpoint へ自動永続化）です。

backend/app/tools/ui_table.py 
from langchain_core.tools import tool
from app.agent.ui import ui  # 薄いヘルパー(任意)

@tool(response_format="content_and_artifact")
def render_table(title: str, columns: list[str], rows: list[list[str]]):
    """表データを対話的テーブルとして表示する。"""
    # 第1要素(content)=LLM文脈用の要約、第2要素(artifact)=UI封筒
    return ui("table", summary=f"{title}: {len(rows)}行の表を表示",
              title=title, columns=columns, rows=rows)

利便性のための薄いヘルパー（必須ではありません）。UIEnvelope 組み立ての定型を吸収するだけです。

backend/app/agent/ui.py 
def ui(component: str, *, summary: str, mode: str = "declarative", **props):
    """UIツールの戻り値 (content, artifact) を組み立てる。"""
    return summary, {"v": UI_SCHEMA_VERSION, "kind": "ui",
                     "component": component, "mode": mode, "props": props}

summary(content)の役割と規約: artifact は LLM に見えません（content のみ後続文脈へ渡る）。要約が貧弱だとモデルが「何を表示したか」を後続ターンで誤認します。UIツールの content は必ず意味のある日本語要約にすること（規約）。content_to_text は artifact を無視するため、executor のスクラッチパッド要約（step_result）にも影響しません。

MCPの扱い: mcp_servers.json への1行追加哲学は不変です。MCPツールが将来 artifact を返せば同経路でUI化、返さなければ従来テキスト。loader/registry の改修は不要です。

✅

拡張ポイントがそのまま効く

「app/tools/ にファイルを置くだけ」「mcp_servers.json に1行足すだけ」という既存の2大拡張ポイント（拡張ガイド）が、UIでも完全にそのまま機能します。これが artifact 方式を選ぶ核心的な利点です。

新SSEイベント ui_resource と発行経路

新イベント名 ui_resource を1種だけ追加します。tool_result（最大2000字テキスト要約）は据え置き、UIは別イベントで構造化データとして分離します（前方互換）。SSEは順序保証されるため、同一 id に対し tool_call → ui_resource → tool_result の順で届きます。フロントは id で既存 toolCall に ui を後付けできます。

emit のトリガ箇所は backend/app/services/streaming.py の mode=="updates" の ToolMessage 分岐（現状119-135行）に1分岐追加するのみです。関数冒頭に seen_ui: set[str] = set() を新設し、tool_result の yield 直前にUI封筒を流します。

backend/app/services/streaming.py 
elif isinstance(msg, ToolMessage):
    if accumulate:
        turn_messages.append(msg)
    ui = coerce_ui(getattr(msg, "artifact", None), msg.tool_call_id)
    if ui is not None:
        ukey = f"ui:{ui['id']}"
        if ukey not in seen_ui:
            seen_ui.add(ukey)
            yield ServerSentEvent(event="ui_resource",
                                  data={**ui, "name": msg.name})
    # 既存 tool_result はそのまま続行（要約テキストは常に流す）
    key = str(msg.tool_call_id or msg.id)
    if key in seen_tool_results:
        continue
    seen_tool_results.add(key)
    yield ServerSentEvent(event="tool_result", data={...})  # 既存のまま

dedupe: 既存 seen_tool_results（tool_call_id 基準）と同様に seen_ui を ui:{id} キーで新設します。responder が親レベル update（全履歴の再掲）とサブグラフ内 updates で二重配信する既存問題は、ui_resource も既存の ns==() スキップと accumulate ガードと同じ経路を通るため自動的に守られます。ただしテスト（test_streaming_bridge.py）に二重配信ケースの追加が必要です。

前方互換: useChatStream.ts の dispatch() switch は default: break で未知イベントを既に無視しています（114行）。ui_resource ハンドラを追加するまでフロント旧版でも無害です。逆にバックエンド旧版に対しフロント新版も、イベントが来ないだけで安全。done/error 排他、id dedupe の既存規約をそのまま新イベントにも適用します。

SSE として送られる新旧イベントの形は次のとおりです。ui_resource は構造化UIデータ、tool_result は据え置きの要約テキストで、両者は併存します。

eventdata の形向き用途
ui_resource{ id: string(tool_call_id または ui_id), name?: string, component: string, mode: "declarative"|"iframe", props: object, v: number }server→clientツールが返した構造化UIデータ（封筒）をフロントへ届け、id一致の tool_call に後付けして宣言的レンダラで描画させる。tool_result(要約テキスト)とは別イベントで分離し前方互換を維持。tool_call→ui_resource→tool_result の順で同一idに届く
tool_result{ id: string, name: string, content: string(最大2000字) }server→client変更なし（据え置き）。UIツールでも要約テキストは従来通り常に流す。ui_resource と併存し、UI非対応の旧フロントはこれだけ読めば従来描画になる（前方互換）

UI→エージェントの往復経路

SSEは下り一方向のため、ボタン/フォーム操作等の上り経路が別途必要です。2案を比較し、案2を推奨します。

案1（非推奨）新エンドポイント POST /api/chat/ui-action

{thread_id, user_id, ui_id, action, payload} を受け、サーバ側でアクションを ToolMessage 等へ変換して再 invoke します。

短所: グラフへの注入点を新設する必要があり「コア無改修」を破ります。アクション→メッセージ変換の独自プロトコルが増え、決定論制御の表面積が拡大。checkpoint との整合（どの状態に追記するか）が複雑になります。

案2（推奨）chat/stream へ構造化メッセージ再投入

UI操作を ChatRequest.message に載せ、既存の POST /api/chat/stream を再利用します（APIリファレンス）。

  • フロントは操作を人間可読＋機械可読な1メッセージに整形して送る。例: フォーム送信 → message = '[ui-action] form#<ui_id>: {"name":"..."}'。
  • バックエンドは新コード不要。orchestrator が自然言語/JSON混在メッセージとして処理（parsing.py の extract_json が JSON 部を堅牢抽出）。
  • スレッド継続・checkpoint・履歴・リフレクションすべて既存経路に乗る。シングルユーザv1・決定論重視に最も合致。

揺れの抑制（任意の後方互換拡張）: ChatRequest に ui_action: dict | None = None を既定 None で追加してよい。旧クライアント無影響です。これがあれば streaming.py 冒頭で if body.ui_action: message = format_ui_action(...) と定型整形でき、フロントの文字列整形をサーバへ寄せて非決定性を抑えられます。

MVP実装: ChatRequest 無改修で「整形済みメッセージを send() に渡す」だけ。UIResource の onAction が ChatView の handleSend(formatUiAction(...)) を呼ぶ最小実装で成立します。

app限定アクションの概念（将来）: MCP Apps の visibility:app 相当の「モデル文脈に出さないUI専用アクション」を導入し、リフレッシュ/ページング/フォーム送信がエージェント文脈やマルチエージェントのトークンを汚さないようにする余地を残します。

⚠️

非決定性に注意

案2は LLM が ui-action メッセージを自然言語として再解釈するため、整形が緩いと非決定的になり得ます。ChatRequest.ui_action 構造化フィールド＋streaming 冒頭の定型整形で揺れを抑えるのが望ましい。

永続化と履歴リロード時の再水和

永続化（追加実装ほぼ不要）: UIリソースは ToolMessage.artifact に入り、AsyncPostgresSaver の checkpoint にメッセージの一部として自動保存されます（データモデルと永続化）。別テーブル・別保存経路は不要。これが artifact 方式を選ぶ最大の利点です。

MessageOut を後方互換で拡張します。

backend/app/schemas/chat.py 
class MessageOut(BaseModel):
    ...
    ui: dict[str, Any] | None = None   # 後方互換 (既定 None)

threads.py で response_model_exclude_none=True を指定済みのため、UIなしメッセージのレスポンスは一切変化せず前方互換です。

履歴復元の再水和は _to_api_message() の ToolMessage 分岐に追加します。判定は streaming.py と同じ coerce_ui を共有し、ライブstreamとリロード復元の完全一致を保証します。

backend/app/services/history.py 
if isinstance(msg, ToolMessage):
    out = {"role": "tool", "content": _content_to_text(msg.content),
           "id": msg.id, "tool_call_id": msg.tool_call_id, "name": msg.name}
    ui = coerce_ui(getattr(msg, "artifact", None), msg.tool_call_id)
    if ui is not None:
        out["ui"] = ui
    return out

フロント toUIMessages（ChatView.tsx 19-54行）: tool メッセージ処理で m.ui があれば、突合先の UIToolCall に ui を後付け（現状 call.result = m.content と同じ箇所）。孤立 tool でも対応します。これでリロード後もUIが再描画されます。

UIローカル状態の保持（将来）: 軽い状態（タブ選択・展開状態等）は ui_id をキーに localStorage。ユーザの労力がかかる状態（フォーム下書き等）は LangGraph checkpoints や LangMem store へ app限定アクション経由で保存。OpenAI の setWidgetState 相当を自前提供する設計余地を残します。

ℹ️

executor 経由UIの注意

executor サブグラフ（checkpointer=False、親 messages 非共有）内でUIツールが呼ばれると ToolMessage が永続化されません。MVPでは UIツールは高速パス responder 経由でのみ確実に永続化される点に留意してください。executor 経由UIは synthesizer のテキスト要約に縮約されます。

React描画とレジストリ

まず型を追加します。SSE ui_resource の data 形と MessageOut.ui を共通の UIResource で表します。

frontend/src/api/types.ts 
export interface UIResource {
  id: string; name?: string; component: string;
  mode: "declarative" | "iframe";
  props: Record<string, unknown>; v: number;
}
export type UIResourceEvent = UIResource;  // SSE ui_resource の data

MessageOut に ui?: UIResource を、UIToolCall（ToolCallChip.tsx）に ui?: UIResource を追加します。

SSEハンドラ（useChatStream.ts）: StreamHandlers に onUIResource?: (e: UIResourceEvent) => void を追加、dispatch switch に case "ui_resource" を追加。ChatView の handleSend に onUIResource を実装し、updateMessage で該当 assistantId の toolCalls 内（id一致）に ui を後付け、無ければメッセージ直下の uiResources 配列へ。

新コンポーネント（UIResource.tsx）: component 名でホワイトリスト解決する宣言的レンダラ。未知 component は ToolCallChip にフォールバック（前方互換＝コア無改修のフロント版）。

frontend/src/components/UIResource.tsx 
const REGISTRY: Record<string, React.FC<{ props: unknown; onAction: (a: unknown) => void }>> = {
  table: TableView, chart: ChartView, card: CardView, form: FormView,
};
export function UIResource({ res, onAction }: { res: UIResource; onAction: (a: unknown) => void }) {
  if (res.mode === "iframe") return <SandboxFrame res={res} onAction={onAction} />; // 将来
  const C = REGISTRY[res.component];
  if (!C) return <ToolCallChip call={{ id: res.id, name: res.name ?? res.component, args: {} }} />;
  // props は各 View 内で Zod 検証してから使う
  return <div className="ui-resource"><C props={res.props} onAction={onAction} /></div>;
}

MessageBubble.tsx: tool-chips 描画の隣で、ui を持つ toolCall は ToolCallChip でなく UIResource を描画（チップは details で内訳、UIは展開表示）。react-markdown のテキストとGUIパートは MessageBubble 内で共存させます。onAction は ChatView 経由で前述の往復経路（案2 handleSend(formatUiAction)）へ接続します。

props 検証（coding-style 準拠）: 各 View は Zod スキーマで props を parse してから描きます。検証失敗時はフォールバック表示。これにより MCP由来の無検証 props による XSS/プロトタイプ汚染を防ぎます。

追加依存: chart は recharts 等（既存 react-markdown と同様の追加ライブラリ）。フロント全体の構成はフロントエンド設計を参照してください。

セキュリティ

MVP（A 宣言的方式）の安全性

  • 任意HTML/JSを一切実行しない。component 名はホワイトリスト解決のみ、未知名はフォールバック。XSS面はゼロ。
  • props は各 View 内で Zod 検証してから使う。MCP/LLM由来の不正データ・プロトタイプ汚染を防ぐ。
  • content(要約2000字相当)と props(UIデータ)の分離で、巨大データがLLM文脈へ混入するコスト/安全境界を維持。
  • props 肥大化対策: table の rows 等が大きいと SSE/checkpoint/履歴APIが肥大する。tool_result の2000字制限に相当する上限（行数/バイト数）を props にも設け、超過時はサーバ側でページネーション or 要約する（要検討事項）。

将来（B iframe方式）に導入するセキュリティ機構（MCP Apps 基準、実装は後回し）

  • 二重iframeサンドボックス: ホスト→外側プロキシiframe(別オリジン)→内側iframe(untrusted UI)。FastAPI に別オリジン相当のサンドボックス配信エンドポイント（例 GET /sandbox/proxy）を追加し、外側が postMessage を双方向中継・検証する。
  • sandbox属性: 内側iframeは allow-scripts のみから始め、必要分だけ allow-forms 等を足す。allow-same-origin は隔離を弱めるため原則付与しない（別オリジン＋別iframeで補償）。
  • 隔離の能動検証: window.top アクセスが SecurityError を投げることで隔離成立を確認する。
  • オリジン検証: 受信側は event.origin が想定ホストオリジンと一致するか必ず確認し、想定外を破棄。返信先も明示originで送りワイルドカード禁止。
  • CSP: metaタグより改竄耐性の高いHTTPヘッダでFastAPIが付与。restrictive-by-default（default-src 'none'）とし、connect/resource/frame/baseUri の各ドメインを宣言的に許可。未宣言なら外部接続ゼロ＝ネットワーク遮断が既定。
  • ストレージ遮断: Claude Artifacts 流に localStorage/sessionStorage をブロック。通信は postMessage越し JSON-RPC のみで全往復を監査可能にし、UIはホストDOM/Cookie/ストレージへ直接アクセス不可とする。
  • 権限明示: 必要なら camera/microphone/geolocation/clipboard を iframe allow 属性で個別許可（MCP の meta.ui.permissions 相当）。
⚠️

無認証ローカルv1では B方式は割に合わない

無認証ローカルv1では、任意HTML/JS実行（B方式）は LLM/MCP由来の悪性スクリプトという攻撃面を持ち込み割に合いません。MVPは A方式に限定し、封筒に mode フィールドだけ予約して B の実装は後回しにするのが安全な既定方針です。

ライフサイクル全体（生成から再水和まで）

UIリソースの一生は次の5段階です。emit（生成）、live render（ライブ描画）、interact（操作で往復）、persist（永続化）、rehydrate（リロード再水和）。ライブstreamとリロード復元はどちらも同じ coerce_ui 純関数を通るため、表示結果が完全一致します。

MVPスコープ

v1 で確実に実装する範囲です。

  • UIリソースの封筒（UIEnvelope: v/kind/component/props、任意 mode/ui_id）を backend/app/agent/ui.py に定義し、is_ui_artifact / coerce_ui 純関数を streaming.py と history.py で共有する
  • UIツールは @tool(response_format="content_and_artifact") で記述し、app/tools/ に置くだけで registry.py の isinstance(BaseTool) 自動登録に乗せる（graph.py/registry 無改修）
  • 薄いヘルパー ui(component, summary=..., **props) を提供（必須ではない利便API）
  • 新SSEイベント ui_resource を streaming.py の ToolMessage 分岐に1つ追加。seen_ui で ui:{id} dedupe、tool_result は据え置きで両方流す
  • 宣言的セーフコンポーネント方式(A)のみ実装: フロントに REGISTRY(table/chart/card/form)、props は Zod 検証、未知 component は ToolCallChip フォールバック
  • 封筒に mode フィールド("declarative"|"iframe")を予約。iframe 方式(B)は実装せずデフォルト declarative
  • UI→エージェント往復は案2（chat/stream へ [ui-action] 整形メッセージ再投入）。新エンドポイントは作らない
  • 永続化は ToolMessage.artifact 経由で checkpoint へ自動保存。MessageOut.ui(既定None) と history.py の coerce_ui で履歴リロード時に再水和
  • useChatStream.ts に onUIResource / case "ui_resource" 追加（default:break による未知イベント無視の前方互換を維持）
  • UIツールの content は意味のある日本語要約にする規約（artifact はLLM文脈に渡らないため）

将来拡張

v1 では予約・設計余地を残すだけで、後から非破壊で追加する範囲です。

  • (B) サンドボックスiframe方式: mode:"iframe" の任意HTML/JS を 二重iframe + sandbox属性 + CSP(HTTPヘッダ) + オリジン検証 + postMessage越しJSON-RPC で隔離描画（MCP Apps 2026-01-26 準拠）
  • FastAPI に別オリジン相当のサンドボックス配信エンドポイント（例 GET /sandbox/proxy）を追加
  • 外部MCPサーバの ui-scheme リソース宣言を mcp_servers.json 経由で取り込み、coerce_ui で本封筒へ正規化
  • app限定アクション（visibility:app 相当）の導入で、リフレッシュ/ページング/フォーム送信をエージェント文脈・マルチエージェントのトークンから隠す
  • UIローカル状態の保持: 軽い状態は ui_id キーで localStorage、労力がかかる状態は checkpoints/LangMem へ（setWidgetState 相当の自前API）
  • 部分描画（体感遅延低減）: 既存 custom progress 透過機構で部分 props をストリームし、ローディングプレビューを先出し（ontoolinputpartial / input-streaming 相当）
  • ChatRequest.ui_action 構造化フィールド＋streaming 冒頭の format_ui_action 定型整形で、往復の非決定性を抑制
  • props 肥大化対策: ui props に行数/バイト数上限を設け、超過時はサーバ側ページネーション or 要約
  • executor サブグラフ経由UIの永続化（現状は responder 高速パスのみ確実に永続化）

未解決の論点

確定仕様に対して、実装着手前に決め切りたい論点を残しています。

  • UIツールの content(要約)を「規約」で縛るだけで十分か、それとも ui() ヘルパーで summary 必須化＋lint等の機械的強制を入れるべきか
  • props のサイズ上限値（行数/バイト数）の具体値と、超過時の挙動（ページネーション vs 要約 vs 切り詰め）
  • 往復経路は MVP で ChatRequest 無改修（フロント文字列整形）で出すか、最初から ui_action 構造化フィールドを入れて非決定性を抑えるか
  • chart コンポーネントの追加依存（recharts 等）の採否とバンドルサイズ影響
  • executor 経由でUIツールが呼ばれた場合に synthesizer 要約へ縮約される挙動を MVP で許容するか、responder 経路に明示誘導するか
  • ui_resource の二重配信 dedupe を tool_call_id 基準にした際、ui_id 明示指定との優先順位（同一ツールが複数UIを返すケース）
  • 履歴 API の props 肥大時に GET /api/threads/{id}/messages のレスポンス上限/ページングをどう扱うか
