システムアーキテクチャ

このシステムが「どんな部品で」「どうつながって」動いているのか、そして「なぜこの構成を選んだのか」を、実際のソースコードに即して解説します。全体像をつかめば、以降の各ページ（マルチエージェント設計・記憶・ストリーミングなど）がぐっと読みやすくなります。

このプロジェクトは、ローカルで動く LLM（大規模言語モデル＝文章を生成するAIの本体）である Ollama を頭脳に使う、チャット型の汎用AIエージェントです。認証はなく、v1 はシングルユーザー（利用者を識別する user_id は "default-user" 固定）で動きます。ここでは、画面（フロントエンド）・サーバー（バックエンド）・データベース・LLM の4つがどう協調するかを見ていきます。

全体構成（コンテナ図）

まずは「箱（プロセス）」のレベルで全体像を見ます。登場するのは4つの実行プロセスです。ブラウザの中で動く フロントエンド、Python で書かれた バックエンド、データを保存する PostgreSQL、そして AI 本体の Ollama です。フロントエンドとバックエンドは別々のポートで動くため、開発時はフロントの開発サーバー（Vite）が /api 宛のリクエストをバックエンドへ中継（プロキシ）します。

ポイントは、バックエンドが中心（ハブ）になっていることです。フロントエンドは PostgreSQL にも Ollama にも直接アクセスしません。すべての通信はバックエンドの /api を経由します。これにより、AI への問い合わせ方やデータの保存方法を変えても、フロントエンド側の修正を最小限にできます。

ℹ️

ポイント

ポート番号は、フロント開発サーバーが 5173、バックエンドが 8000、PostgreSQL が 5432、Ollama が 11434 です。PostgreSQL は docker-compose.yml で pgvector/pgvector:pg17 イメージを使って起動します（ベクトル検索のため、ただの PostgreSQL ではなく pgvector 入りが必須です）。

バックエンドのディレクトリ構成

バックエンド（backend/app/）は、役割ごとにフォルダを分けています。「画面からのリクエストを受ける層」「AI の思考ロジック層」「裏方の処理層」がきれいに分かれているのが特徴です。次の図はフォルダの親子関係を表したものです。

各フォルダの役割は次の表のとおりです。実際のファイル名に対応しています。

フォルダ / ファイル役割
routers/HTTP リクエストの入口。chat.py(チャット), threads.py(会話スレッド), memory.py(記憶), health.py(死活監視), tools.py(ツール一覧) の各エンドポイントを定義。
agent/AI の思考エンジン本体。graph.py(エージェント構築), state.py(状態), routing.py(分岐判断), parsing.py(出力解析), prompts.py(指示文), models.py(LLM 定義), context.py(文脈)。
agent/nodes/マルチエージェントを構成する各役割ノード。orchestrator・planner・executor・evaluator・synthesizer と共通処理 common。executor / evaluator は依存（depends_on）が解決済みのステップ群をラウンド単位で並列実行・並列評価する。詳しくは マルチエージェント設計。
services/裏方の処理。streaming.py(SSE 配信), history.py(会話履歴の取得), threads.py(スレッド操作)。
memory/長期記憶まわり。manager.py(記憶の抽出・反映), tools.py(LangMem のツール), store_query.py(記憶の検索)。記憶アーキテクチャ参照。
mcp/loader.py。外部ツールサーバー（MCP）の設定を読み込み、ツールとして取り込む。拡張ガイド参照。
tools/エージェントに渡すネイティブツール群。registry.py が calculator・datetime_tool・web_search などの @tool を集める。

schemas/リクエスト・レスポンスの型（chat.py など）。入出力の形をはっきり決める。

起動時に何が起きるか（lifespan）

FastAPI には lifespan（アプリの起動から終了までを管理する仕組み）があり、このプロジェクトは main.py の lifespan 関数で初期化処理を1か所に集約しています。サーバー起動時に一度だけ、DB 接続プールを作り、永続化レイヤを準備し、エージェントを組み立てます。順を追って見てみましょう。

実際のコード上の流れは次のとおりです。

  1. build_pool(settings.database_url) で 共有の接続プールを1つ作り、await pool.open() で開きます。
  2. init_persistence(pool, settings) が、そのプールの上に checkpointer（会話履歴を DB に保存する仕組み）と store（長期記憶＋ベクトル検索の保管庫）を作り、それぞれ setup() を一度だけ実行してテーブルを用意します。
  3. ensure_threads_table(pool) で、スレッドのメタ情報（タイトルや更新日時）を保存する threads テーブルを作ります。これはチェックポインタには含まれない独自テーブルです。
  4. build_registry() でネイティブツールを集め、langmem_hotpath_tools(store) で記憶ツールを用意し、get_mcp_tools() で外部ツール（MCP）を取り込みます。
  5. build_agent(...) で、上記すべてのツール・checkpointer・store を渡してエージェント本体を組み立てます。
  6. build_reflection_executor(store, settings) で、会話の後に記憶を整理する ReflectionExecutor を用意します。
  7. これらを app.state（アプリ全体で共有できる置き場）に格納し、yield でリクエスト受付を開始します。
✅

堅牢な後片付け

lifespan は try / finally で囲まれており、起動が途中で失敗しても finally で必ず後片付けします。順序は「reflection_executor.shutdown() → pool.close()」で、バックグラウンド処理を止めてから接続プールを閉じます。

ℹ️

テスト用の早期リターン

設定 skip_startup（環境変数 APP_SKIP_STARTUP=1）が真のときは、PostgreSQL・Ollama・MCP に接続せず、pool や agent を None にしたまま起動します。DB や LLM を用意できないテスト環境でアプリを立ち上げるための仕組みです。

MCP ツール取得の失敗耐性

外部ツールサーバー（MCP）の読み込みは、失敗してもアプリ全体が落ちないように設計されています。get_mcp_tools() の呼び出しは try / except で囲まれており、例外が起きてもログを残してネイティブツールだけで処理を続行します。外部依存があってもエージェントの基本機能が止まらない、という安全設計です。

技術スタックと選定理由

採用している主な技術と、なぜそれを選んだのかを表にまとめます。設計の4本柱（拡張性・ストリーミング・user_id の貫通・シンプルさ）に沿った選定になっています。

領域採用技術選定理由

マルチエージェント用の追加設定

config.py には、上の基本設定に加えてマルチエージェントグラフ用の設定（control_model、計画ステップ上限 max_plan_steps、再計画上限 max_replans、親グラフの graph_recursion_limit など）も定義されています。これらは無限ループ防止やコンテキスト切り詰めのための値で、詳細は マルチエージェント設計で扱います。

DB 接続の方針

このシステムは、checkpointer・store・エージェントの3者が同じ1つの接続プールを共有します。プールごとに接続を持つのではなく、起動時に build_pool() で作った AsyncConnectionPool を全員で使い回す設計です。接続数を抑え、初期化も1か所にまとまるので、シンプルさという設計方針に合致します。

プールを作るときの接続オプション（kwargs）には、次の3つを必ず指定しています。これは LangGraph の Postgres 永続化が正しく動くための必須設定です。

backend/app/core/db.py 
def build_pool(database_url: str) -> AsyncConnectionPool:
    """単一の共有 AsyncConnectionPool を作る (open は呼び出し側で await)。"""
    return AsyncConnectionPool(
        conninfo=database_url,
        open=False,
        kwargs={
            "autocommit": True,
            "prepare_threshold": 0,
            "row_factory": dict_row,
        },
    )
接続オプション役割
autocommit: True各文を即座にコミット。LangGraph の Postgres セーバー／ストアが前提とする動作。
prepare_threshold: 0プリペアドステートメントの自動準備を無効化。プール越しの再利用で起きる不整合を防ぐ。
row_factory: dict_rowクエリ結果を辞書（カラム名→値）で受け取る。
⚠️

落とし穴: from_conn_string は使わない

LangGraph のチュートリアルでよく出てくる from_conn_string は、内部で接続を開いて自動的に閉じてしまうため、このプロジェクトでは使いません。接続が閉じられると、共有プールを使い回す前提が崩れてしまうからです。代わりに、自前で作った AsyncConnectionPool を AsyncPostgresSaver(pool) / AsyncPostgresStore(pool, ...) に渡し、ライフサイクルを lifespan 側で一元管理しています。上の kwargs 3点を欠かすと永続化が正しく動かないので、プール生成時に必ず指定してください。
