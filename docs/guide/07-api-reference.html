API リファレンス

バックエンド（FastAPI）が提供する全エンドポイントの「唯一の真実」です。各APIについて、説明・リクエスト・レスポンス（JSON例）をセットで示します。フロントエンドはここに書かれた仕様だけを根拠に通信します。

ℹ️

ポイント

このプロジェクトは認証なしのシングルユーザー構成です（v1）。ユーザーを区別する user_id はフロントエンドが固定値 "default-user" を送ります。user_id がどの層を貫通するかは 記憶アーキテクチャ も参照してください。

共通仕様

まず全エンドポイントに共通するルールを押さえておきましょう。個別仕様を読むときの前提になります。

  • ベースパス: すべての API は /api/... から始まります。開発時、フロントエンド（Vite 開発サーバー、ポート 5173）は相対パス /api/... を呼び、Vite の proxy（リクエストを別サーバーへ転送する仕組み）がバックエンド（ポート 8000）へ中継します。
  • Content-Type: SSE（後述の POST /api/chat/stream）以外のレスポンスは application/json です。リクエストボディを送る POST も Content-Type: application/json を付けます。
  • user_id: 多くの API がクエリパラメータ user_id を受け取ります。省略時のデフォルトは "default-user" ですが、フロントは明示的に送ります。
  • エラー形状: 4xx / 5xx のエラーは FastAPI 標準の形 {"detail":"..."} で返ります。たとえば存在しないスレッドを操作すると 404 {"detail":"thread not found"} です。
⚠️

日時の表記

レスポンス中の created_at / updated_at は UTC の ISO 8601 形式で、末尾が Z になります（例 2026-06-14T09:00:00Z）。バックエンドのシリアライザがこの形に統一しています。

GET /api/threads

そのユーザーが持つ会話スレッド（1つの会話のまとまり）の一覧を返します。並び順は更新日時の降順（最近やり取りしたものが先頭）です。

リクエスト

パラメータ場所必須説明

GET /api/threads?user_id=default-user

レスポンス（200）

{

リクエストボディ

フィールド型必須説明

POST /api/threads
Content-Type: application/json

{
  "user_id": "default-user",
  "title": null
}

レスポンス（201）

✅

タイトルは後から自動で付く

ここで title を省略しても、最初のチャット送信時にユーザーメッセージの先頭から自動でタイトルが設定されます（詳細は POST /api/chat/stream の項を参照）。

GET /api/threads/{thread_id}/messages

指定スレッドの会話履歴（過去のメッセージ列）を取得します。画面を再読み込みしたときなどに、それまでのやり取りを復元するために使います。履歴は checkpointer（会話履歴をDBに保存する仕組み）から読み出されます。

リクエスト

GET /api/threads/8f3c0e2a-1b4d-4f6a-9c2e-7a1b2c3d4e5f/messages?user_id=default-user

スレッドが存在しない、または別ユーザーのものなら 404 {"detail":"thread not found"} を返します。

レスポンス（200）

メッセージは role によって含まれるフィールドが変わります。値が無いフィールドはレスポンスから省略されます（response_model_exclude_none による）。

role意味追加フィールド
userユーザーの発言—
assistantエージェントの返答tool_calls（ツール呼び出しがあれば [{id, name, args}]）
toolツール実行の結果tool_call_id, name
 
{
  "thread_id": "8f3c0e2a-1b4d-4f6a-9c2e-7a1b2c3d4e5f",
  "messages": [
    { "role": "user", "content": "東京の天気は?", "id": "msg-1" },
    {
      "role": "assistant",
      "content": "",
      "id": "msg-2",
      "tool_calls": [
        { "id": "call-1", "name": "get_weather", "args": { "city": "Tokyo" } }
      ]
    },
    {
      "role": "tool",
      "content": "晴れ、最高28度",
      "id": "msg-3",
      "tool_call_id": "call-1",
      "name": "get_weather"
    },
    { "role": "assistant", "content": "東京は晴れ、最高28度です。", "id": "msg-4" }
  ]
}

まだ発言が無いスレッドでは "messages": [] になります。

リクエスト

レスポンス（200）

  "deleted": true
}

SSE POST /api/chat/stream

このプロジェクトの中心となる API です。ユーザーのメッセージをエージェントに渡し、応答を SSE（Server-Sent Events）で少しずつ（トークン単位で）流し返します。SSE とは、1本の HTTP 接続でサーバーがイベントを順次送り続ける仕組みです。SSE 全体の設計思想は ストリーミング設計 を参照してください。

リクエスト

通常の JSON で送りますが、レスポンスは SSE で受けるため Accept ヘッダで宣言します。

ヘッダ値
Content-Typeapplication/json
Accepttext/event-stream
フィールド型必須説明
messagestring必須ユーザーの入力テキスト
thread_idstring必須作成済みのスレッドID
user_idstring任意（既定 "default-user"）ユーザー
 
POST /api/chat/stream
Content-Type: application/json
Accept: text/event-stream

{
  "message": "東京の天気は?",
  "thread_id": "8f3c0e2a-1b4d-4f6a-9c2e-7a1b2c3d4e5f",
  "user_id": "default-user"
}
🛑

スレッドは事前作成が必須

thread_id は POST /api/threads で作っておく必要があります。未知のIDを渡すと、SSE のバイトを1つも送出する前に通常の HTTP 404 {"detail":"thread not found"} が返ります。これはルータの依存解決（_ensure_thread）でストリーム開始前にチェックしているためで、フロントは「SSE 開始前のエラー」と「SSE 中のエラーイベント」を区別して扱えます。

ℹ️

タイトルの自動設定と keep-alive

そのスレッドで最初のユーザーメッセージのとき、ストリーミング開始前に先頭40字がタイトルとして設定されます。また接続維持のため、15秒ごとに :（コロン）で始まる keep-alive コメント行が送られます。コメント行はイベントではないのでパース時は無視します。

SSE で送信される処理の流れ

クライアント送信からイベント受信までの全体像です。404 はストリーム開始前、それ以降は SSE イベントとして届く点に注目してください。

SSE イベントの種類

SSE は event: 行でイベント名、data: 行で本文（JSON）を運びます。本 API が送るイベントは次の4種類です。フィールド名は厳密にこの通りです。

eventdata の中身意味
token{content, node}応答テキストの断片。node は生成元で responder または synthesizer
tool_call{id, name, args}エージェントがツールを呼び出した
tool_result{id, name, content}ツールの実行結果。content は最大2000字に切り詰め
progress{status, ...}進捗通知。status は必須、その他は拡張フィールド（任意）

progress イベントの拡張フィールド

progress は status 以外に状況を表す任意フィールドを持てます。すべて optionalで、状況に応じて一部だけ付きます。

フィールド型 / 取りうる値意味
phaserouting / plan / step / evaluate / synthesize現在の処理フェーズ
plan[{id, description}]立案された計画ステップ一覧
step / totalnumber何番目／全何ステップか
verdictpass / retry / replan / fail評価フェーズの判定結果
done{thread_id, title}完了通知（最終的なスレッド情報）
error{message}エラー発生時のメッセージ
ℹ️

phase / verdict はマルチエージェントの内部状態

これらは計画・実行・評価を行うマルチエージェント処理の進捗を表します。各フェーズの意味は マルチエージェント設計 で詳しく解説しています。

ワイヤ形式（実際に流れるバイト）

SSE は1行ずつのテキストです。event: 行と data: 行（JSONは1行）が続き、空行でそのイベントが確定します。: で始まる行は keep-alive コメントです。実際のバイト列は次のようになります。

event: progress
data: {"status":"working","phase":"routing"}

event: tool_call
data: {"id":"call-1","name":"get_weather","args":{"city":"Tokyo"}}

event: tool_result
data: {"id":"call-1","name":"get_weather","content":"晴れ、最高28度"}

event: token
data: {"content":"東京","node":"responder"}

event: token
data: {"content":"は晴れ","node":"responder"}

: keep-alive

event: progress
data: {"status":"done","done":{"thread_id":"8f3c0e2a-...","title":"東京の天気は?"}}

フロント側のパース規約

フロントは fetch + ReadableStream でこのバイト列を読み、次の手順で1イベントずつ復元します。

  1. TextDecoder({stream:true}) で受信チャンクを UTF-8 として結合する（文字が途中で分割されても安全に連結する）。
  2. event: 行が来たらイベント名を覚える。
  3. data: 行が来たら本文として蓄積する。
  4. 空行が来たらそこまでをひとつのイベントとして確定し、data を JSON パースして処理する。
  5. : で始まる行（keep-alive コメント）は無視する。

GET /api/memory

MEMORY

そのユーザーの長期記憶（会話をまたいで覚えておく情報）を一覧・検索します。query の有無で動作が変わります。

パラメータ場所必須説明
user_idクエリ任意（既定 "default-user"）対象ユーザー
queryクエリ任意省略時は更新が新しい順。指定時はセマンティック検索（意味の近さで検索）
limitクエリ任意（既定 20）取得件数の上限
 
GET /api/memory?user_id=default-user&query=好きな食べ物&limit=20

レスポンス（200）

各記憶の content は、ストア（記憶を保存する場所）の値から取り出した人間が読めるテキストです。namespace は記憶の所属を表し ["memories", user_id] の形です。score は検索時のみ値が入り（意味の近さ）、一覧時（query 省略）は null です。

{
  "user_id": "default-user",
  "memories": [
    {
      "key": "mem-001",
      "content": "ユーザーはラーメンが好き",
      "namespace": ["memories", "default-user"],
      "updated_at": "2026-06-10T12:00:00Z",
      "score": 0.82
    }
  ]
}
✅

score の有無で UI を切り替える

フロントは score が null かどうかで「一覧表示」と「検索結果（関連度付き）」を見分けられます。

POST /api/memory/forget/preview

MEMORY

「仕事の話は全部忘れて」のようなスコープ付き一括忘却の第1段階です。query に意味の近い記憶を検索し、削除候補を返します。このエンドポイントは何も削除しません（非破壊）。実際の削除は、ユーザーが候補を確認したうえで /api/memory/forget/confirm を呼びます。誤削除を防ぐ確認ゲートの一部です。

POST /api/memory/forget/preview
Content-Type: application/json

{
  "query": "仕事",
  "user_id": "default-user",
  "limit": 20
}

レスポンス（200）

candidates は削除候補です。返る件数は limit（既定 20）で決まり、さらに confirm 側で最大 DEFAULT_MAX_KEYS（25 件）まで（大量削除の安全弁）に制限されます。各候補には意味検索の score が付きます。

{
  "candidates": [
    {
      "key": "mem-007",
      "content": "残業が多い職場で働いている",
      "score": 0.88,
      "updated_at": "2026-06-10T12:00:00Z"
    }
  ]
}

POST /api/memory/forget/confirm

MEMORY

一括忘却の第2段階です。ユーザーが承認した keys を実際に削除し、削除後に各 key を aget で直接照会して Absence 検証（消したはずの記憶が本当に消えたか）を行います。意味検索の top-k に依存しないため、関連記憶が多くても取りこぼしません。keys は安全弁として最大 25 件まで（超過は 422）。

POST /api/memory/forget/confirm
Content-Type: application/json

{
  "keys": ["mem-007"],
  "user_id": "default-user"
}

レスポンス（200）

verified が true なら全 key の消去を確認できた状態です。false のときは leaked_keys に「まだ残っていた key」が列挙され、フロントは「一部の記憶が残っている可能性」を警告できます。

{
  "deleted_count": 1,
  "verified": true,
  "leaked_keys": []
}
🛑

confirm は不可逆・必ず確認を挟む

/forget/confirm は記憶を物理削除します（復元不可）。フロントは必ず /forget/preview → ユーザー確認 → /forget/confirm の順で呼びます。エージェント主導の forget_memories ツールも同じ2段階プロトコル（confirm=false で候補提示 → 承認 → confirm=true）に従います。詳細は 記憶アーキテクチャ を参照してください。

レスポンス（200）

フィールド説明

スレッドのライフサイクル

最後に、これらの API がどう連なって1つの会話を成立させるかを状態遷移で確認します。スレッドは「作成 → メッセージ送受信 → 削除」という流れをたどります。

CORS と開発環境

開発時はフロント（ポート 5173）とバックエンド（ポート 8000）がポート違いで動くため、ブラウザの CORS（別オリジンへの通信制限）への対応が必要です。本プロジェクトの設定は次の通りです。

  • バックエンドは CORSMiddleware を入れ、allow_origins=["http://localhost:5173"] でフロント開発サーバーからのアクセスを許可します。
  • フロントは相対パス /api/... を呼び、Vite の proxy が :8000 へ転送します。これによりフロントのコードにバックエンドの絶対URLを埋め込まずに済みます。
✅

フロントは絶対URLを書かない

常に /api/... の相対パスで呼ぶのが規約です。環境ごとの転送先は Vite proxy（開発）や本番のリバースプロキシ側で吸収します。セットアップ手順は セットアップと運用 を参照してください。
