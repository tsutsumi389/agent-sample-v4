フロントエンド設計

このページでは、React 19 + TypeScript + Vite で作られた画面の構成と、コンポーネントの親子関係・状態の持ち方・SSE（サーバーから少しずつ届くデータ）の受信処理を、実際のソースコードに沿って解説します。これを読めば「どのファイルが何を担当しているか」が一通り把握できます。

前提として、このアプリは認証なしのシングルユーザー（user_id は固定値 "default-user"）で動くチャット型 AI エージェントです。フロントエンドはバックエンドの REST API と SSE エンドポイント（POST /api/chat/stream）だけを呼び出し、画面を組み立てます。API の詳細は APIリファレンス、ストリーミングの設計全体は ストリーミング設計 を参照してください。

画面構成（3ペイン）

画面は左から右へ 3 つの領域（ペイン）に分かれています。これは最上位コンポーネント App が <div className="app"> の中に 3 つのコンポーネントを横並びで配置することで作られています。

ペインコンポーネント役割
左Sidebar会話スレッドの一覧表示・新規作成・削除。クリックで表示するスレッドを切り替える。
中央ChatView選択中スレッドの会話表示と、メッセージ送信・SSE 受信によるリアルタイム表示。
右MemoryPanelAI が長期記憶として覚えている内容（メモリ）の一覧・検索・削除・一括忘却。
ℹ️

ポイント

「スレッド」とは 1 つの会話のまとまり（履歴を束ねる単位）です。スレッドを切り替えると、その会話の過去メッセージが中央ペインに読み込まれます。

コンポーネント・ツリー

各コンポーネントの親子関係は次の図のとおりです。App を頂点に、3 つのペインがぶら下がります。中央の ChatView はさらに、1 件ずつのメッセージを描く MessageBubble（その中でツール実行を表示する ToolCallChip を使う）と、入力欄の Composer を子に持ちます。

ファイルの場所は frontend/src/ 配下にまとまっています。最上位は App.tsx、画面部品は components/、状態ロジックは hooks/、API 通信は api/ に分かれています。

状態管理とデータの流れ

このアプリは外部の状態管理ライブラリを使わず、React 標準の useState とカスタムフック（再利用できる状態ロジックの部品）だけで状態を持ちます。「どの状態を誰が持つか」を整理すると次のようになります。

状態持ち主説明
activeThreadIdApp現在表示中のスレッド ID。null なら未選択。
memoryRefreshKeyApp数値カウンタ。増えるたびに MemoryPanel が記憶を再取得する合図。
threads / loadinguseThreadsスレッド一覧と読み込み状態。CRUD（作成・取得・削除）操作もこのフックが提供。
messages / progress / errorChatView表示中スレッドのメッセージ配列、進捗テキスト、エラー文言。
streaminguseChatStreamSSE 受信中かどうか。送信・停止の関数もこのフックが返す。
memories / queryMemoryPanel記憶の一覧と検索キーワード。

状態は「上位の App が共有が必要なものだけを持ち、各ペインに props として渡す」設計です。たとえば「どのスレッドを開いているか」は Sidebar（一覧の選択表示）と ChatView（履歴の読み込み）の両方が必要なので、App が activeThreadId を持って両方に渡しています。

useThreads — スレッドの CRUD

useThreads(userId) は frontend/src/hooks/useThreads.ts にあり、スレッド一覧の取得と作成・削除をまとめたフックです。返す値は次のとおりです。

  • threads: スレッド配列、loading: 読み込み中フラグ、error: エラー文言。
  • refresh(): 一覧を再取得する。
  • create(): 新規スレッドを作って一覧を更新し、作った Thread を返す。
  • remove(threadId): スレッドを削除して一覧を更新する。

マウント時（最初の表示時）に useEffect で refresh() を自動実行し、一覧を読み込みます。create と remove は内部で refresh を呼ぶため、操作後の一覧は常に最新になります。

api/client.ts と api/types.ts

api/client.ts は fetch の薄いラッパです。共通関数 request<T> が「失敗時はレスポンスの detail を取り出して Error を投げ、成功時は JSON を返す」という処理を一手に引き受けます。各 API はこの関数を呼ぶだけの短い関数として並びます。

関数HTTP用途
listThreads(userId)GETスレッド一覧を取得
createThread(userId, title?)POSTスレッドを新規作成
getMessages(threadId, userId)GETスレッドの履歴メッセージを取得
deleteThread(threadId, userId)DELETEスレッドを削除
listMemories(userId, query?, limit)GET記憶を一覧・検索（既定 limit=20）
deleteMemory(key, userId)DELETE記憶を 1 件削除
forgetPreview(userId, query, limit)POST一括忘却の候補を取得（非破壊）
forgetConfirm(userId, keys)POST承認した記憶を一括削除し検証
listTools()GET利用可能なツール一覧を取得

すべての関数が userId をクエリパラメータやボディに含めており、user_id がフロント側から API まで一貫して流れる設計（設計4本柱の「user_id の貫通」）になっています。なお SSE の送信だけはこのラッパを通さず、後述の useChatStream が直接 fetch します。

api/types.ts は API 契約（api_contract.md）と対応する TypeScript の型定義です。Thread / MessageOut / Memory といったレスポンス型に加え、SSE で届く TokenEvent / ToolCallEvent / ToolResultEvent / ProgressEvent / DoneEvent / ErrorEvent や、送信ボディ ChatStreamRequest が定義されています。

送信からスレッド作成までの流れ

メッセージ送信は ChatView の handleSend が担います。ここで重要なのは「スレッド未選択でも送信できる」点です。その場合は先にスレッドを作ってから送信します。これはバックエンドの契約上、存在しない thread_id へ送ると 404 になるためです。

  1. 送信中（streaming）または送信処理中なら何もしない（二重送信防止）。
  2. threadId が null なら createThread(userId) でスレッドを作り、onThreadCreated(thread) で App に通知。App はその ID を activeThreadId にセットする。
  3. ローカルに「ユーザー発言」と「空のアシスタント発言」を即座に追加して、入力直後の体感速度を上げる。
  4. useChatStream の send() を呼び、SSE 受信を開始する。
⚠️

スレッド切替とストリームの取り違え防止

送信時に「ストリームを所有するスレッド ID」を streamingThreadIdRef に記録し、各イベントハンドラの先頭で owns()（今表示中のスレッドが所有者か）を確認しています。これにより、ストリーム中に別スレッドへ切り替えても、届いたトークンが別の会話に紛れ込まないようになっています。

SSE 受信の流れ（useChatStream）

frontend/src/hooks/useChatStream.ts は、SSE を fetch + ReadableStream で受信する手書きパーサです。ブラウザ標準の EventSource は GET 専用で POST やカスタムヘッダが使えないため、ここでは fetch でストリームを開き、自前で行を解析しています。次の図が一連の流れです。

パーサの内部処理を分解すると次のようになります。

  • UTF-8 の結合: TextDecoder("utf-8") を {stream:true} で使い、日本語などのマルチバイト文字がチャンク境界で分断されても正しく結合します。
  • 行分割とフィールド解釈: バッファを \n で区切り、各行を processLine が解釈します。event: 行はイベント名、data: 行はデータ本文として蓄積。: で始まる行は keep-alive コメントとして無視し、\r\n（CRLF）にも対応します。
  • 空行でディスパッチ: 空行を 1 イベントの区切りとみなし、dispatch() が蓄積した data 行を JSON としてパースし、イベント名に応じたハンドラへ渡します。

イベント名ごとの扱いと、それを受けた ChatView 側の画面反映は次のとおりです。

イベントハンドラ画面での反映
tokenonTokenアシスタント発言の content に文字を追記（少しずつ本文が伸びる）。
tool_callonToolCallツール呼び出しチップを追加（「実行中」表示）。
tool_resultonToolResult同じ id のチップに結果を入れる（「完了」表示）。
progressonProgressprogress 状態を更新し、進捗テキストを表示。
doneonDone進捗をクリアし、App の handleDone へ。一覧更新と記憶再取得を起動。
erroronError進捗をクリアしエラー文言を表示。
🛑

send は必ずイベントハンドラから呼ぶ

ソースのコメントにあるとおり、send を useEffect から呼んではいけません。React 19 の StrictMode は開発時に effect を二重実行するため、effect 起点だと送信が 2 回走ってしまいます。このアプリでは「送信ボタンのクリック」というユーザー操作を起点にしています。

停止は stop() が AbortController の abort() を呼ぶことで実現します。中断は AbortError として握りつぶされ、エラー扱いにはなりません。また送信中に abortRef が埋まっている間は send の再入を拒否し、二重ストリームを防ぎます。done で受け取った最終イベントの内容（DoneEvent = thread_id と title）をきっかけに、App がスレッド一覧を refresh し（タイトルが確定する）、memoryRefreshKey を増やして記憶パネルを再取得します。

会話の表示（MessageBubble と履歴の変換）

MessageBubble は 1 件のメッセージを吹き出しで描きます。送信者（あなた / アシスタント）の表示、本文、そしてツール呼び出しがあればチップを並べます。ストリーミング中で本文がまだ空のときは「考え中...」、本文が出始めたら点滅カーソルを表示して、生成中であることを示します。

履歴 API から取得したメッセージは、ChatView 内の toUIMessages が表示用モデル（UIMessage）へ変換します。ここでのポイントは、role: "tool" のメッセージ（ツールの実行結果）を独立した吹き出しにせず、対応する呼び出しに tool_call_id で突き合わせて、アシスタント発言の中のチップへ畳み込むことです。これにより「呼び出しと結果が 1 つのチップにまとまる」見た目になります。

ツール呼び出しの可視化（ToolCallChip）

ToolCallChip は、AI がツール（外部機能）を使ったことを <details> の開閉ボックスで見せるコンポーネントです。要約部分にはツール名と状態バッジ（結果がまだなら「実行中」、届いたら「完了」）を表示し、開くと引数（JSON を整形表示）と結果が見られます。

ストリーミング中は、先に tool_call が来てチップが「実行中」で現れ、後から tool_result が同じ id で届いて「完了」に変わる、という流れになります。ツール実行がイベントとしてどう流れてくるかは ストリーミング設計、各ツールの API 仕様は APIリファレンス を参照してください。native や MCP といったツールの種別については 拡張ガイド で扱います。

AI の記憶パネル（MemoryPanel）

MemoryPanel は、AI が長期記憶として覚えている内容を一覧・検索・削除する右ペインです。refreshKey という props が増えるたびに load() を呼んで再取得します。App は会話の done ごとにこのキーを増やすため、会話のたびに記憶表示が更新されます。検索欄にキーワードを入れて検索すると、ベクトル検索のスコア（関連度）が各項目に表示されます。記憶の仕組みそのものは 記憶アーキテクチャ を参照してください。

各記憶の「削除」ボタンは 1 件ずつの削除です。これに加えて、検索欄の下の「この検索結果をまとめて忘れる」ボタンでスコープ付き一括忘却ができます。押すと forgetPreview で削除候補を取得し、確認モーダルを開きます（この時点では削除しません）。モーダルでは候補がチェックボックス付きで並び、外せば残せます。「N件を忘れる」を押すと forgetConfirm が実削除と検証を行い、結果をパネル上部の通知に表示します。検証で verified=false のときは「一部の記憶が残っている可能性」を警告します。削除は不可逆なので、この確認モーダルは必須の安全装置です（設計は 記憶アーキテクチャ、API は APIリファレンス を参照）。

✅

記憶は少し遅れて反映される

パネルにも注記があるとおり、会話内容が記憶に統合されるのはバックグラウンド処理で 30 秒ほどかかります。直近の発言がすぐ記憶に出てこなくても正常です。

Vite 開発サーバと proxy

開発時はフロント（Vite、ポート 5173）とバックエンド（FastAPI、ポート 8000）が別ポートで動くため、ブラウザから直接 8000 番を叩くと CORS（別オリジンへのアクセス制限）に引っかかります。これを避けるため、vite.config.ts の server.proxy で /api へのリクエストをバックエンドへ中継しています。

frontend/vite.config.ts 
export default defineConfig({
  plugins: [react()],
  server: {
    port: 5173,
    proxy: {
      "/api": {
        target: "http://localhost:8000",
        changeOrigin: true,
      },
    },
  },
});

この設定のおかげで、フロントのコードは常に /api/... という同一オリジン相対パスだけを書けばよく、開発環境の URL の違いを意識せずに済みます。実際 api/client.ts と useChatStream.ts はすべて /api 始まりのパスを使っています。開発サーバの起動手順は セットアップと運用 を参照してください。

ℹ️

user_id の決まり方

App.tsx では import.meta.env.VITE_USER_ID ?? "default-user" で user_id を決めています。環境変数 VITE_USER_ID が無ければ既定値 "default-user" が使われ、v1 のシングルユーザー前提と一致します。

まとめ

フロントエンドは、App が 3 ペインと共有状態を束ね、スレッド操作は useThreads、送信と SSE 受信は useChatStream という 2 つのフックに分離されたシンプルな構成です。SSE は標準 EventSource ではなく fetch + ReadableStream による手書きパーサで、POST・カスタムヘッダ・中断（停止ボタン）に対応します。関連する設計は ストリーミング設計、APIリファレンス、記憶アーキテクチャ を合わせて読むと全体像がつかめます。
