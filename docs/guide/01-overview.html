はじめに — 全体像と用語

このページは設計書の入口です。このプロジェクトが「何で」「何ができて」「どんな部品でできているか」を、地図のように一望できるようにまとめました。専門用語はその場で一言ずつ説明するので、プログラミングが初めての方でも読み進められます。

このエージェントは何？

このプロジェクトは、自分のPCの中で動くLLM（大規模言語モデル＝文章を生成・理解するAI）を頭脳にしたチャットボットです。チャット画面に質問を打ち込むと、AIが文章で答えてくれます。

ふつうのチャットボットと違うのは、ただ会話するだけではない点です。このエージェントは必要に応じて「ツール（AIが呼び出せる小さな機能。計算・日時取得・Web検索など）」を使って調べたり計算したりできます。さらに、会話の中で出てきたあなたの好みや事実を長期記憶として覚えておき、次回以降の会話に活かします。

そしてもう一つ大事なのが、すべてローカル（自分の環境）で完結することです。AIモデルは Ollama（手元のPCでLLMを動かすためのソフト）で動かすので、会話の内容を外部のクラウドAPIに送りません。プライバシーを保ちながらAIを使えます。

ℹ️

ポイント

このv1（最初のバージョン）は認証なしのシングルユーザーを前提にしています。利用者を識別する user_id は "default-user" という固定値が使われます。将来の複数ユーザー対応に備えて、設計のあちこちに user_id を通す「貫通」の仕組みが入っています（後述の4本柱を参照）。

何ができるか（ユースケース例）

このエージェントが実際に役立つ場面を、具体例で見てみましょう。

  • 調べもの: 「東京の明日の天気は？」のような質問に、Web検索ツールで最新情報を調べて答える。
  • 計算: 「1500円の品物を3つ、消費税10%込みでいくら？」のような計算をツールで正確に処理する。
  • 日付・時刻: 「今日は何曜日？」「来週の月曜は何日？」など、現在日時を取得して答える。
  • 一般的な質問応答: ツールを使わずに、AI自身の知識で文章を組み立てて返す。
  • あなたを覚える: 「私はコーヒーより紅茶が好き」と伝えておくと、それを長期記憶に保存し、後日「飲み物を勧めて」と聞いたときに紅茶を提案する。

どのツールを使うか、あるいはツールを使わず自分で答えるかは、AIがその場で判断します（この判断のしくみは「ReAct」と呼ばれます。後述の用語集で説明します）。

設計の4本柱

この設計には、ぶれない4つの方針（柱）があります。ページのどこを読んでいても、この4つに立ち返れば「なぜこうなっているか」が理解できます。

柱意味
① 拡張性（最重要）新しいツールや MCP（外部ツール群への共通接続規格）を、エージェントの中心コードを書き換えずに追加できる。機能を増やしても本体が壊れにくい。
② ストリーミングAIの返答を一文字ずつ流れるように表示する。SSE（サーバーから少しずつデータを送る仕組み）を使い、待ち時間を体感的に短くする。
③ user_id の貫通「誰の会話か」「誰の記憶か」を表す user_id を、フロントからDBまで一本の線で通す。これにより将来の複数ユーザー対応がスムーズになる。
④ シンプルさ余計な抽象化を避け、標準的な部品を素直に組み合わせる。読みやすく、追いやすい構造を保つ。
✅

迷ったら拡張性

4本柱の中でも特に重視されているのが「拡張性」です。詳しい作法は拡張ガイドにまとめられています。

登場人物と用語集

この設計書には、たくさんの部品とカタカナ用語が登場します。先にここで一覧にしておきます。すべて覚える必要はありません。「この単語なんだっけ？」と思ったらここに戻ってきてください。

用語やさしい説明
LLM大規模言語モデル。文章を理解し、続きの文章を生成するAI。このエージェントの「頭脳」。
Ollama手元のPCでLLMを動かすためのソフト。これがあるおかげで会話を外部に送らずに済む。
gpt-ossOllamaで動かすLLMの1つ。チャットの返答とツール呼び出しを担う主役（コンテキスト長 num_ctx=32768、推論の深さ reasoning="medium"）。
qwen3もう1つのLLM。制御の判断や記憶の抽出など、裏方の小さな判断を担う（num_ctx=8192）。
nomic-embed-text文章を埋め込み（768次元の数値ベクトル）に変換するモデル。文章の「意味の近さ」を計算でき、記憶の検索に使う。
LangGraphエージェントの処理の流れを「グラフ（点と線の図）」として組み立てる枠組み。このプロジェクトの土台。
create_agentLangGraph（langchain.agents.create_agent）が用意する、エージェントを手軽に作る関数。これを呼ぶと標準的なエージェントが組み上がる。
ノード / エッジ / StateGraphノード＝処理の一段階、エッジ＝段階どうしのつながり、StateGraph＝それらをまとめて「状態（State）」を持ち回る図。LangGraphの基本部品。
checkpointer（短期記憶）会話履歴をDBに保存する仕組み。これにより会話の続きを覚えていられる。ここでは AsyncPostgresSaver を使う。
Store / pgvector（長期記憶のDB）Store＝長期記憶の置き場所。pgvector＝PostgreSQLでベクトル検索を可能にする拡張。ここでは AsyncPostgresStore（vector768）を使う。
LangMem会話から「覚えておくべき事実」を抽出し、長期記憶に保存・検索するためのライブラリ。
MCPModel Context Protocol。外部のツール群にAIをつなぐための共通規格。これに対応していれば、いろいろなツールを統一的な作法で追加できる。
ツールAIが呼び出せる小さな機能。計算・日時取得・Web検索など。AIは状況に応じてツールを選んで使う。
SSEServer-Sent Events。サーバーからクライアントへデータを少しずつ送り続ける仕組み。返答のストリーミング表示に使う。
ReAct「考える（Reason）→行動する（Act）」を繰り返すAIの動き方。考えてツールを使い、結果を見てまた考える、を必要なだけ繰り返す。
namespace記憶を整理するための「引き出しのラベル」。user_id ごとに分けることで、誰の記憶かを区別する。
user_id利用者を識別する値。v1では "default-user" 固定。会話履歴や記憶を「誰のものか」で紐づける鍵。

登場部品の関係を図で見る

用語だけだとつながりが見えにくいので、主要な部品が互いにどう関係するかを図にしました。AIの「頭脳」であるLLMを中心に、記憶の置き場所やツールがぶら下がっているイメージです。

この図のとおり、エージェントは1種類のLLMだけでなく、役割に応じて gpt-oss と qwen3 を使い分けます。記憶には「会話の続きを覚える短期記憶（checkpointer）」と「人物の事実を覚える長期記憶（Store）」の2系統があります。それぞれの詳しい作りは記憶アーキテクチャで説明します。

1つの質問が流れる様子

では、あなたが質問を1つ送ったとき、裏側で何が起こるのでしょうか。高いところから全体の流れを眺めてみます。細かい部品はあとで掘り下げるので、ここでは「だいたいこう動く」をつかめれば十分です。

注目してほしいのは、返答が完成してから一気に表示されるのではなく、生成されたそばからSSEで少しずつ送られてくる点です。これが4本柱の「②ストリーミング」です。仕組みの詳細はストリーミング設計にあります。また、ツールを使うかどうかをAIが判断して繰り返す動き（alt の部分）が「ReAct」です。

全体像

最後に、システム全体の構成を1枚にまとめます。大きく分けると「フロント（画面）」「バックエンド（頭脳と司令塔）」「PostgreSQL（記憶の保管庫）」「Ollama（LLM実行）」「ツール・MCP（道具）」の5つです。

使われている技術を一覧にすると次のとおりです。

層主な技術
フロントReact 19 + TypeScript 5.9 + Vite 8。SSEは fetch と ReadableStream で受信。
バックエンドPython 3.12 + uv + FastAPI 0.136 + uvicorn。エージェント基盤は LangGraph 1.x。
LLM 実行Ollama（gpt-oss / qwen3 / nomic-embed-text）。
データベースPostgreSQL（pgvector/pgvector:pg17）。会話履歴は checkpointer、長期記憶は Store + vector768。

各層の役割分担や設計判断の理由はシステムアーキテクチャでくわしく説明します。

この先の読み方

この設計書は、概念から具体へ少しずつ降りていく構成です。目的に合わせて、次の道案内を参考にしてください。

  1. システムアーキテクチャ — 全体の層構造と、各部品がどう責任を分け合うかを把握する。まずはここから。
  2. マルチエージェント設計 — 複数のLLM・役割をどう組み合わせて1つのエージェントにしているか。
  3. 記憶アーキテクチャ — 短期記憶（checkpointer）と長期記憶（Store + LangMem）のしくみ。
  4. ストリーミング設計 — 返答を少しずつ届けるSSEの実装。
  5. 拡張ガイド — コアを触らずにツールやMCPを追加する作法（4本柱の最重要）。
  6. APIリファレンス — フロントとバックエンドをつなぐ各エンドポイントの仕様。
  7. データモデルと永続化 — DBに何がどう保存されるか。
  8. フロントエンド設計 — 画面側の構成とSSE受信の組み立て。
  9. セットアップと運用 — 実際に動かす手順と運用のポイント。
✅
